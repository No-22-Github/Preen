## 0. 产品定位

Preen 是一台 RWKV-7 state tuning 的工作台。目标用户是**已经知道 state tuning 是什么、
或者愿意花十分钟搞清楚**的人：模型爱好者、独立开发者、想给自己的角色卡训一个人格的人。

这类用户的共同特征是：**不怕复杂，怕不透明**。他们能读懂 loss 曲线，
但没有耐心去 `man` 一个参数；他们想知道机器现在到底吃了多少内存，
而不是看一个转圈的进度条。

界面因此有两个职责，优先级从高到低：

1. **可观测** —— 训练在发生什么、机器扛不扛得住、这个 state 到底有没有用，
   都必须在界面上直接看到，不需要开活动监视器、不需要去翻日志文件。
2. **好上手** —— 默认值就是能跑的值，主动线不需要读文档。

第二条不能以牺牲第一条为代价。

## 1. 设计立场

### 1.1 原生 ≠ 遮蔽

macOS 应用的"低门槛"来自**默认值可信**与**路径清晰**，不是来自**削减能力**。
参照系是 Xcode / Instruments / Console.app / TablePlus / Tower —— 这些应用把
非常底层的东西（call stack、query plan、reflog）直接摊在界面上，同时依然是
标准的、克制的、可预测的 macOS 应用。

具体到本产品：

- **不把 CLI 已有的能力藏起来。** `--cache-limit-gb`、`--seed`、`--resume`、
  重复惩罚三件套、`--template` / `--reasoning` / `--think`、`doctor` 的环境报告 ——
  这些今天只在 CLI 里存在，它们**都应该在图形界面里有位置**。位置可以是二级
  （Inspector、设置、折叠区），但不能是"没有"。
- **不发明抽象。** 不把 state 叫"人格档案"，不把 checkpoint 叫"存档点"。
  用户在 GitHub issue 里、在 RWKV 群里、在本仓库文档里看到的是同一个词。

### 1.2 渐进披露，不是分级隐藏

三层：

| 层 | 内容 | 交互成本 |
|---|---|---|
| 一层 | 主动线：选数据 → 训练 → 对话验证 → 导出 | 零点击可见 |
| 二层 | 超参、采样参数、会话状态、事件日志 | 一次点击（折叠区 / Inspector / ⌥⌘I） |
| 三层 | 环境自检、MLX 缓存上限、数据目录、Python 路径 | 设置窗口 |

判据：**任何一个 CLI flag，都能在三次点击内找到对应的图形控件。**
找不到的，要么补上，要么在本文里写明"故意不做"及理由。

### 1.3 Preen 不是 CLI 的 GUI 外壳

它们共用 `service` / `serve` 这一层，但**面向的不是同一件事**：
CLI 面向脚本化与可复现（`--json`、事件流、退出码）；GUI 面向观测与迭代
（曲线、A/B、随手改一个参数再跑一遍）。

因此**不做**"复制等效 CLI 命令"这类功能 —— 它暗示图形界面是命令行的第二等公民，
也会把 CLI 的参数名当作用户模型强加给界面。要脚本化的人本来就会用 CLI。

## 2. 术语约定

**原则**：术语跟随 RWKV 社区与本仓库既有文档（AGENTS.md / Spec / CLI）。
只有当中文译名**既清晰、又不影响社区沟通**时才翻译。不发明新词，不造抽象。

| 场景 | 界面用词 | 说明 |
|---|---|---|
| state / state tuning | **原词** | 这是产品的核心概念，也是社区通用词。不翻译，不包装 |
| loss / prompt / token / seed / temperature / top-p | **原词** | 行业通用，翻译反而增加沟通成本 |
| `.pth` / `.npz` / checkpoint | **原词** | 文件格式就是文件格式。按钮写「导出 .pth」，不写「导出给 Runner」 |
| epoch | **轮** | 中文清晰表意，且不影响社区沟通（"第 3 轮" 无歧义） |
| step | **步** | 同上 |
| learning rate / warmup / early stopping | **学习率 / warmup / 早停** | 前后两个是既有标准中译；warmup 无好中译，保留 |
| held-out | **验证集** | 标准中译，比"held-out"更通用 |
| baseline（无 state） | **无 state（基线）** | 与 CLI 的 A/B 输出用词一致 |
| with_state | **有 state** | 同上 |
| ctx_len | **ctx_len（单条最长 token 数）** | 主标签用原词，副标签给一句人话 |
| template = qa / instruction / raw | **原词 + 一行说明** | 下拉项：`qa — 对话问答` / `instruction — 指令` / `raw — 原样` |
| reasoning + think | **reasoning 模型 / 思考：off · fast · on** | 保留档位原词，旁边一行说明各档行为 |
| cache 续传 / 重放 | **续传 / 重放** | 见 §6.3：这是**要暴露**的信息，不是要隐藏的实现细节 |

**辅助文案的写法**：标签用术语，副标签用人话，一句话说清"改了会怎样"。

> `ctx_len` **512**
> 单条样本最长 token 数。超长会被截断；16 GB 机器上建议 ≤ 512。

不做 tooltip-only 的解释（鼠标悬停才出现的信息等于不存在）。

## 3. 骨架

单窗口，`NavigationSplitView` 两栏 + **常驻底部状态栏**：

```
┌──────────────┬─────────────────────────────────────────────────┐
│  SIDEBAR     │  ┌── toolbar ─────────────────────────────────┐ │
│  (~200pt)    │  │ 标题                      [主操作]  [⋯]    │ │
│              │  └────────────────────────────────────────────┘ │
│  训练         │                                                 │
│  对话         │  主内容                                          │
│  State 库     │                                                 │
│              │                                                 │
│  ──────────  │                                                 │
│  模型         │                                                 │
│  g1d-0.4b ▾  │                                                 │
├──────────────┴─────────────────────────────────────────────────┤
│ ● RSS 11.7 GB · 压力 绿 · 3.4 s/步 · 第 7/20 轮 · loss 2.41 ↓  │  ← §5
└────────────────────────────────────────────────────────────────┘
      最小 1000×680 · 默认 1180×760
```

**模型选择器钉在侧边栏底部**，而不是每个面板各选一次。
理由是工程事实而非美学：`serve` 是单进程单模型（Spec §3.1），换模型 = 重启 sidecar；
且连续 `load_model` 会累积 Metal 内存池（AGENTS.md）。模型是**环境**，不是参数，
它的位置必须体现这一点。切换时给一次确认（会中断当前会话与训练）。

不做标签页、不做多窗口。一次只跑一个重任务 —— 这同样是 AGENTS.md 里的内存事实，
界面不该让用户轻易违反它。

## 4. 训练

### 4.1 四个状态，四个视图

```
[空] ──选数据→ [配置] ──开始→ [运行中] ──→ [完成 / 失败 / 已停止]
                 ↑                              │
                 └──────────── 再训一个 ─────────┘
```

**空**：拖拽区。副标题给出支持格式（JSONL / JSON / CSV；Alpaca、ShareGPT、
ChatML、裸 QA 自动探测）。

**配置**：上下两段。

- **上半（主角，约 2/3）：导入预览。**
  探测结果（schema + confidence）+ 前 3 条样本的**渲染预览**。
  渲染必须做 **token 级 prefix/target 着色**，分界严格取 `RenderedSample.prefix_len`
  （Spec §4.4 已定必做）。这一屏把"loss 只算 target 段"变成用户能看见的东西 ——
  它是本产品最好的教学时刻，也是最好的 bug 发现器。
  探测失败（`schema=unknown`）时降级为字段映射器：左列列出文件里实际存在的键，
  右列两个下拉（prompt ← / response ←），下方实时重渲染预览。

- **下半：超参。** 默认折叠成一行摘要
  （`lr 0.01 · ctx_len 512 · 20 轮 · 早停 patience 3 · seed 42`），点开是完整 Form。
  Form 里必须包含 CLI 有的全部：`lr` / `lr_floor` / `warmup` / `ctx_len` / `epochs` /
  `grad_clip` / `early_stop` / `patience` / `test_ratio` / `checkpoint_every` / `seed` /
  `template`。每个字段偏离默认值时，旁边出现一个"恢复默认"的小箭头。

  > 默认值来自 `decision-precision.md` 与 P0 实测（`lr=1.0` 会让 state 爆炸）。
  > Form 不应给出会炸的取值范围而不加提示：`lr > 0.1` 时给一条 inline warning。

**运行中**：见 §5。loss 曲线与资源曲线共轴，是这个视图的全部。

**完成**：产物卡片（state 路径、层数、最终 loss、跑了几轮、耗时、`data_sha256` 前 8 位）
+ 两个按钮：**「去对话」（primary）**、「导出 .pth」。
「去对话」必须直接把这个 state 装进对话面板并跳转 —— 这是产品最重要的一条动线。

**已停止**：保留曲线，给「从 checkpoint 继续」（对应 `--resume`）。
checkpoint 目录里的每个 `epochN.npz` 都应可见、可选、可续。

### 4.2 事件 → 界面映射

`events.py` 的事件类型与界面元素一一对应。**Swift 侧 enum 必须穷举，不许 `default`** ——
新增事件类型时编译期就要炸出来。

| 事件 | 界面反应 |
|---|---|
| `start` | 进入「运行中」，用 `config` 快照填顶部摘要 |
| `epoch_start` | 轮次计数 +1 |
| `step` | loss 曲线追加一点（按 `log_every` 抽样，不做平滑） |
| `epoch_end` | 画一条轮分隔线；有 `held_out_loss` 时画第二条曲线（虚线） |
| `checkpoint` | 事件日志一行；checkpoint 列表追加一项。不打断 |
| `early_stop` | 曲线上标一面旗；文案「已停在第 N 轮（后续没有更好）」 |
| `std_warning` | **产品默认不启用**（阈值未标定，见 AGENTS.md）。收到即忽略 |
| `final` | Trainer 算完但产物**尚未落盘**。显示"收尾中"，**不要显示完成** |
| `completed` | 唯一可信的完成信号。切「完成」态 |
| `failed` | 切失败态，`message` 原样显示 + 事件日志自动展开 |
| `cancelled` | 切「已停止」，保留曲线 + 提供 resume |

### 4.3 事件日志

一个可折叠的抽屉（默认收起），逐行显示原始事件流 —— 与 `--events-file` 同源的 JSON lines。
失败时自动展开。

这不是"CLI 的 GUI 版"，这是**可观测性**：训练是长任务，失败必须能自证。
参照 Xcode 的 Console 抽屉。支持筛选（type）、复制、导出。

## 5. 性能监视

**需求原话**：训练跑起来后把窗口丢到后台，回来时**打开窗口 3 秒内**要能知道
"电脑怎么样了、训练怎么样了"。这是本产品的一个一等功能，不是附属品。

### 5.1 采样：Swift 侧做，不打扰训练循环

**不新增协议指令，不改 Python。** Swift 侧对 sidecar 子进程直接采样：

- **进程 RSS**：`proc_pid_rusage(pid, RUSAGE_INFO_V4)` → `ri_resident_size`。
  这是真 RSS。**绝不显示 `mx.get_peak_memory()`** —— AGENTS.md 明文
  「严重漏报…永远以 RSS 为准」，显示一个仓库文档明令不可信的数字，
  比不显示更坏（见附录 A2）。
- **系统内存压力与换页**：`host_statistics64` / `vm_statistics64` 的
  `compressions` / `swapins`。**换页是真正的性能悬崖**，它比内存数字本身重要。
- **吞吐**：由 `step` 事件的时间戳差分算出 s/步。
  它是内存压力最早的可观测指标（AGENTS.md 的 ms/step 拐点法就靠它）。

采样 1 Hz。**单位一律 GB（÷1e9），禁止 GiB** —— AGENTS.md 的单位铁律，界面同样受约束。

### 5.2 常驻状态栏（全局，任何面板可见）

```
● RSS 11.7 GB / 12.07 GB 削顶线 · 压力 绿 · 3.4 s/步 · 第 7/20 轮 · loss 2.41 ↓ · 剩余 ~18 min
```

- 左侧圆点是**唯一的颜色语义**：绿（正常）/ 黄（RSS 越过削顶线 90%，或 s/步 较基线劣化 >30%）/
  红（发生换页）。
- 削顶线 = `max_recommended_working_set_size × 95%`，取自 `doctor` 报告
  （AGENTS.md 红线标定：16 GB 机器约 12.07 GB）。
- 空闲时状态栏只显示机器状态，训练时才追加训练状态。

### 5.3 双轨共轴图（训练「运行中」视图的主体）

```
loss   ┤ ╲                                          ← 上轨：train loss（实线）
       ┤  ╲╲___                                        + held-out loss（虚线）
       ┤       ╲╲______
       └────────────────────────────────────────
RSS    ┤          ╭──────────────  ← 削顶线（红色水平虚线）
       ┤      ╭───╯                                 ← 下轨：sidecar RSS
       ┤ ╭────╯          ▓ 换页（红色竖条）
       └────────────────────────────────────────
        0        200       400       600   步
```

**两轨共用同一时间/步轴**。这是整个监视功能的核心价值：
把 loss 抖动、s/步 劣化、RSS 顶穿削顶线、换页事件**在同一根轴上对齐**，
一眼能看出"loss 那一下抖是因为换页了"还是"真的训崩了"。
活动监视器做不到这件事 —— 它不知道你的 step 在哪。

鼠标悬停十字线，同时读出该步的 loss / RSS / s-per-step / lr。

### 5.4 后台可见性

「丢后台去刷手机」是明确的使用场景，因此：

- **Dock 图标进度条**（`NSDockTile.progress`）：不切回窗口也能看到进度。
- **系统通知**：`completed` / `failed` / `cancelled` 各发一条，
  `failed` 的通知正文带 `message` 第一行。
- **不做**菜单栏常驻图标（一个训练工具没有资格常驻菜单栏）。

### 5.5 为什么监视器不是一个独立面板

因为独立面板意味着"要点一下才能看到"，直接违反 3 秒判据。
监视信息应当**跟着用户走**（状态栏），并在最需要它的地方**就地展开**（训练视图的共轴图）。
新增一个侧边栏项，只会让用户在两个面板之间来回切。

## 6. 对话

面板名叫**「对话」**而不是「预览」：它本来就是一个 session-stateful 的多轮会话
（Spec §2 已经把 cache 续传/重放做完了），A/B 只是它的一个模式。
未来若长成完整的本地推理前端（见 §10），名字不用改。

```
┌──────────────────────────────────────────────────────────────────┐
│ state: nekoqa_04b_s42.npz ▾   思考: off ▾   [A/B] ⌘D    [⋯] ⌥⌘I │
├───────────────────────────┬──────────────────────────────────────┤
│ 无 state（基线）           │ 有 state · nekoqa_04b_s42            │
│                           │                                      │
│ 你好，有什么可以帮你的？    │ 喵～主人今天想做什么呀？               │
│ ▌                         │ ▌                                    │
│                           │                                      │
│ [stop=eos · 42 tok ·      │ [stop=eos · 38 tok ·                 │
│  38.1 t/s]                │  37.6 t/s]                           │
├───────────────────────────┴──────────────────────────────────────┤
│ ┌────────────────────────────────────────────┐        [发送 ⏎]   │
│ └────────────────────────────────────────────┘                   │
└──────────────────────────────────────────────────────────────────┘
```

### 6.1 A/B 默认开

第一次打开对话面板就是双栏。理由：产品的核心命题是"state tuning 有用"，
要求用户先发现一个开关才能看到证据，等于把论证成本转嫁给用户。
且 README 头图 GIF（Spec §7.1）拍的就是这一屏 —— **默认态即取景框**。
⌘D 收起为单栏，供日常聊天。

两栏同 seed、同配置、同时流式。视觉上只用一个信号区分：右栏一条 2pt accent 左边框。
每栏底部保留 `summary_line()` 的技术摘要（stop_reason / token 数 / t/s）—— 
这是有用信息，不是噪音，dim 显示即可。

### 6.2 参数全部可达

Inspector（⌥⌘I）里放齐：`temperature` / `top_p` / `max_tokens` / `seed` /
`presence_penalty` / `frequency_penalty` / `penalty_decay`，每项标注 ChatRWKV 官方默认值
（`chat.py::config_groups` 已经这么做了，界面照搬）。
主界面只留 `state` 选择器、`思考` 档位（仅当模型是 reasoning 系时显示，
由 `hello` 的 `capabilities` 决定）、A/B 开关。

`template`（qa / instruction / raw）放在 Inspector 顶部，改动会重置会话 —— 
这一点要在界面上说明（换模板 = 换编码格式）。

### 6.3 会话状态要暴露，不要隐藏

Inspector 底部一个「会话」区：

```
轮数        3
本轮路径     续传（cache 干净）
cache       clean
state       nekoqa_04b_s42.npz
```

`续传 / 重放` 是**真实且有用**的信息：它解释了为什么某一轮的首 token 慢了半秒
（重放要重新 prefill 整段历史）。对这个产品的用户来说，这是"哦，原来如此"，
不是"这是什么鬼"。

> 这一条推翻了 Spec §5 里"脏时下一轮打'重建中'标记"的表述 —— 不是因为要隐藏它，
> 而是因为它不该以**一个看不懂的动词**出现在主界面。它属于会话检查器。

`/rewind`（撤销 n 轮）在界面上就是消息 hover 出现的「回到这里」。

### 6.4 思考段

`think=on` 时，`<think>` 段用 dim + hairline 与正式回答分隔，默认折叠为
「思考 12 秒 · 展开」。

⚠️ **工程约束**：当前 `turn_end.result.text` 是**含 `<think>` 标签的原始文本**，
拆分逻辑在 `console.split_thinking`（Python 侧、rich 依赖）。
**Swift 不得重新实现这个拆分** —— 见 §9 · T1。

### 6.5 失败与空态

- 未选 state：右栏显示 State 库里的候选列表 + 一句「选一个 state，看它把模型变成了什么样」。
  空态是入口，不是道歉。
- serve 进程死了：面板顶部一条 banner「推理进程已停止 · 重新启动」，
  点一下重启并**用 history 自动重放恢复会话**（Spec §5 验收 c）。
  不弹 alert，不丢用户已输入的文本。

## 7. State 库

`List`，一行一个 state。行内只显示：文件名、训练数据名、创建时间、最终 loss。
其余全进详情侧栏 —— 即 `metadata.json` 的完整展开：

```
模型         models/converted/rwkv7-g1d-0.4b
数据         nekoqa_smoke_200.json
```

## 8. 视觉与文案

极简的代价是每个细节都要准。

| 项 | 规定 |
|---|---|
| 字体 | 全系统字体。正文 `.body`；模型输出 `.body`（自然语言，**不用等宽**）；token / hash / 路径 / 数值指标 `.system(.caption, design: .monospaced)` |
| 颜色 | 只用语义色：`.primary` / `.secondary` / `.accentColor` / `.red` / `.orange`。**不定义品牌色**，accent 跟随系统偏好 |
| 着色语义 | 全 App 只有两处允许着色：① 导入预览的 prefix/target 分段 ② 状态栏的健康圆点。其余一律无色 |
| 深色模式 | 全用语义色即可白送。进验收 |
| 圆角 / 阴影 | 只用系统控件自带的。不手写 `cornerRadius`，不用阴影 |
| 动画 | 只有三处：面板切换（系统默认）、流式光标、曲线追加点。其余无动画 |
| 图标 | SF Symbols only |
| 数值 | 内存单位一律 GB（÷1e9）；loss 4 位小数；t/s 1 位小数；时间用 `4m12s` 不用 `252.0s` |
| 错误文案 | 说清发生了什么 + 下一步能做什么。「内存不足。把 ctx_len 降到 384 后重试。」而不是「训练失败」 |
| 语气 | 陈述句，句子大写，无感叹号，不道歉，不卖萌 |

## 9. 阻塞项（图形界面无法绕开，须先在 Python 侧解决）

按依赖顺序。详见 `docs/review-checklist.md`。

- **T1 · think 拆分下沉到 UI 中立层。** `split_thinking` 现居 `console.py`（rich 依赖）；
  `chat._display_text` 用 `rfind`、`console.split_thinking` 用 `find`，**已经不一致**。
  要求：拆分逻辑移出皮肤层；`turn_end` 增 `thinking` / `answer` 字段；
  `text_chunk` 增 `phase: "think" | "answer"`。Swift 只负责画。
- **T2 · `serve` 不得 `import cli`。** 现在 `run_serve` 从 `cli` 里拿 `_apply_cache_limit`，
  把 typer 拖进了 sidecar 进程，且该函数出错抛 `typer.Exit`。抽到 `runtime.py`。
- **T3 · `ChatSession` 需要 public API。** `serve` 正在调 `_state_command(["off"])` 这类
  私有方法并从 `reply.lines[0]` 里捞中文提示塞进协议。给 `set_state` / `rewind` / `reset`
  三个公开方法，斜杠命令退化为它们的 parser。
- **T4 · 协议错误码要对得上。** `set_config` 传 `{"temperature": "hot"}` 现在返回
  `internal` + traceback，应为 `bad_request`。界面对二者的处理完全不同
  （internal → 报 bug；bad_request → 在那个输入框画红边）。错误码错了，界面就没法做对。
- **T5 · abort 后会话状态必须一致。** 现在 abort 会留下孤儿 user turn，且 cache 已被
  就地写脏而 `cache_clean` 仍为 `True`。「停止生成」按钮直接踩这条路径。

## 10. 边界

### 10.1 v1 明确不做

- 模型下载器 / HF Hub 浏览器（Spec §0 已定）
- 数据集编辑器 —— 导入是只读的一次性动作
- 多轮 A/B（双 cache 并行）→ v1.5
- 主题 / 配色方案 / 自定义外观
- 欢迎向导 —— 空态本身就是向导
- 「复制等效 CLI 命令」—— 见 §1.3
- 菜单栏常驻图标

### 10.2 生长方向：本地推理前端

**触发条件**：MLX 原生推理性能达到或超过 llama.cpp（届时"在 Mac 上跑 RWKV"
本身就成立为一个独立价值，而不只是 state tuning 的配套）。

那时 Preen 会长出**模型广场 + 完整聊天前端**。这不是 v1 的承诺，
但 v1 的信息架构**不能把这条路堵死**。因此现在就要守住两条：

1. **对话面板从第一天起就是真会话**（history / rewind / 多轮 cache 生命周期），
   不是一个一次性的 preview 小组件。—— Spec §2 已经做到了，别退化。
2. **侧边栏必须能容纳第四项，模型管理必须能从"底部一个 picker"升格为一个面板**，
   而不推翻其余三个面板。—— 当前的两栏 NavigationSplitView 天然满足。

真正的技术障碍在协议层而不在界面层：`serve` 目前是单进程单模型（Spec §3.1），
模型广场意味着多模型热切换。**这一条不必现在解决，但要知道它在那里。**

## 附录：决策记录

**A1 · 术语不做包装层。**
早期草案设计了一张"内部术语 → 用户术语"的映射表（state → 人格、baseline → 原模型）。
**已否决。** 目标用户会在 RWKV 社区、GitHub issue、本仓库文档之间来回切换，
一套只在本 App 里成立的私有词汇会**增加**而不是降低沟通成本。
真正需要翻译的只有那些中译更清晰且社区无歧义的（epoch → 轮）。
降低门槛靠的是**好的默认值 + 一句人话副标签**，不是重命名。

**A2 · 不显示 `mx.get_peak_memory()`，显示真 RSS。**
AGENTS.md 明确该指标严重漏报（不含 Metal wired memory），须以 RSS 为准。
早期草案的结论是"那就什么都不显示"—— **已否决**：用户明确需要一个内置监视器
（现状是盯着活动监视器看内存曲线像股价一样跳）。正确的解法不是隐藏，
而是**显示对的数字**：跨进程采 `proc_pid_rusage` 的 `ri_resident_size`。

**A3 · 监视器不做独立面板。**
独立面板 = 要点一下才能看到 = 违反 3 秒判据。
监视信息跟着用户走（常驻状态栏），并在最需要它的地方就地展开（训练视图的 loss/RSS 共轴图）。

**A4 · loss 与 RSS 必须共轴。**
这是内置监视器相对活动监视器的**唯一**、也是全部的优势：活动监视器不知道你的 step 在哪。
把 loss 抖动、s/步 劣化、换页事件对齐到同一根轴上，才能回答"是训崩了还是换页了"。

**A5 · A/B 默认开双栏。**
产品的核心命题需要被证明，而不是被发现。默认态即 demo 取景框。

**A6 · 单窗口、模型选择器在侧边栏底部。**
serve 单进程单模型 + 连续 `load_model` 累积 Metal 内存池（均为已归因的工程事实）。
这两条决定了"同时开两个模型"在物理上就是坏主意。
界面的职责是让正确的事情显得自然，而不是把危险选项摆在顺手的位置。
