## 1. 任务 1:数值等价性冒烟(✅ 通过,不发散)

**方法**:固定输入,单层 `_wkv7_step_ops` fp32 vs bf16 各跑 273 步(NekoQA smoke200 的 max token),逐步记录 `y` 和 `state` 的相对误差。两档头数(H=16/0.4B, H=32/1.5B)。

**主指标 = global 相对误差**(L1 范数比,稳定)。`worst`(逐元素最大)在参考值接近 0 时会噪声放大,仅作发散早期诊断,不作判据。

| 配置 | y global(全程) | state global(全程) | 趋势 | 判决 |
|---|---|---|---|---|
| 1.5B (H=32) | 0.4% → **1.1%**,step30 后平稳 | 稳定 ~1.0% | 前 30 步达稳态,后 240 步零增长 | PASS |
| 0.4B (H=16) | 0.4% → **1.1%**,step30 后平稳 | 稳定 ~1.0% | 同上 | PASS |

**bf16 递归不发散。** 误差曲线是"快速饱和型"而非"单调增长型"——前 30 步因 bf16 舍入累积达到 ~1% 稳态,之后每步的加法/乘法误差相互抵消,不再放大。这是健康行为。全程无 nan/inf。

> 数据:`data/smoke_15b_h32.json`、`data/smoke_04b_h16.json`

## 2. 任务 2:训练质量 A/B(✅ loss 通过,风格待你判)

**配置完全对齐**:1.5B + NekoQA 200 条 + 2 epoch + seed42 + lr0.01 + nekoqa 模板 + ctx512 + 不早停。两版唯一差异是 patch(fp32 主干 vs bf16 实验),S₀ 在两版都是 fp32 master。

### (a) loss 曲线(逐 step,seed 相同)

逐 step loss 差值在 ±0.2 内随机波动,**无系统性偏移**(不是"bf16 一致偏高/偏低",而是噪声)。关键节点:

| 节点 | fp32 | bf16 | diff | diff% |
|---|---|---|---|---|
| epoch0 末均值 | 2.4730 | 2.4761 | +0.0031 | +0.12% |
| epoch1 末均值 | 1.7183 | 1.7120 | -0.0062 | **-0.36%** |

**末 loss 差 -0.36%,bf16 甚至略低于 fp32**(噪声范围内)。预写判据(loss 差 <2% → 通过)✅。

### (b) state 张量距离(补丁新增判据)

| 指标 | 值 | 解读 |
|---|---|---|
| 整体余弦相似度 | **0.305** | 远未"重合"(补丁预想 >0.99) |
| 整体相对 L2 | **124.9%** | 数值上差很远 |
| 逐层余弦范围 | [0.07, 0.44] | 24 层全 <0.5 |
| 逐层 std 均值 | fp32=0.125 / bf16=0.138 | bf16 略大,梯度确实在更新 |

**这跟 loss 几乎一致是矛盾的——但不是 bug。** 诊断(§4)确认 bf16 梯度正常流回 fp32 master(std 长到 0.138,不是零)。真相是:**state tuning 的 loss landscape 很平坦**,很多不同的 S₀ 都能达到相近 loss。bf16 的 ~1% 前向误差把优化轨迹推向了 loss landscape 里另一个等价解——所以 loss 一样但 state 数值方向不同。

> 补丁预判"state 重合 → 行为大概率过"是反过来的:这里行为(loss/解码)一致,但 state 不重合。说明对 state tuning,**行为 A/B 比 state 距离更可靠的可用性指标**。

### (c) 逐层 std 对比

bf16 的 std 普遍略大于 fp32(均值 0.138 vs 0.125),各层都正常生长(0.10~0.18),无爆炸层、无死层。

### (d) 十问贪心解码 A/B(客观指标全过,风格留你判)

**客观指标(我标的)**:
- 循环检测:fp32 = 0/10,bf16 = 0/10(无循环)
- 自发终止:两版均正常终止(无跑到 max_tokens 上限)
- 两版都是猫娘风格:括号动作 + 喵/主人/炸毛/尾巴

**风格差异(留你判,我只列客观观察)**:
- 两版回答内容不同(几乎每问都不同),但都符合猫娘人设
- bf16 在部分问上想象力略"跳"(如 Q1 提到云朵、Q10 提到沙滩),但这可能是贪心解码对 state 数值差异敏感所致,不一定是退化
- 完整十问输出见 `data/decode_compare.json` 的 `decode_results`,或本报告附录

## 3. 任务 3:内存与速度账(❌ 没脱离削顶区)

**单位说明(重要)**:本报告内存数字统一 **GB(÷10⁹)**。`working_set = 12.71 GB`(= 旧 GiB 口径的 11.84,同一块内存,见 AGENTS.md 新增规矩)。削顶线 = 12.71 × 95% = 12.08 GB。

### 稳态内存对比(1.5B nekoqa200,100 步,全 GB 口径)

| 档位 | precision | active | cache | **active+cache** | 占 working_set | near_capped | ms/step |
|---|---|---|---|---|---|---|---|
| 无限档 | fp32(基线) | 3.10G | 8.97G | **12.07G** | 95% | 是(削顶) | 998 |
| 无限档 | **bf16** | 3.10G | 8.96G | **12.06G** | 95% | 是(削顶) | 824 |
| c4G档 | bf16 | 3.13G | 4.00G | **7.13G** | 56% | 否 | 784 |

### 削顶判定(补丁升级判据)

> 补丁原文:"bf16 版 nekoqa200 无限档的 active+cache 若显著低于 12.07G(比如落到 10G 以下),即证明脱离天花板、拿到的是真稳态;若仍贴着 12G,说明省的量不够。"

**bf16 无限档 = 12.06G,仍顶 95% 削顶线(12.08G)。→ ❌ 没有脱离天花板。**

bf16 比 fp32 在无限档下只"省"了 0.01G cache(8.96 vs 8.97G)——表面看 bf16 一点没省。但这**不是真相**,见 §4 根因诊断。

### 速度

| | fp32 | bf16 | 比 |
|---|---|---|---|
| ms/step (mean) | 923 | 970 | bf16 慢 5%(噪声) |
| ms/step (last10) | 1118 | 967 | bf16 快 13% |

ctx512 下 bf16 **无明显加速**(自检时 ctx128 快 20%,但 ctx512 下被其他开销主导)。MLX 的 bf16 在 Apple GPU 上没有像 CUDA bf16 那样的明确吞吐红利。

## 4. 根因诊断:bf16 确实省了反向图内存,但被削顶机制盖住

表面数据(bf16 无限档 cache 8.96G ≈ fp32 8.97G)会让人误判"bf16 没省内存"。这是削顶机制制造的假象。诊断方法:L=200 单样本,分别测 **前向 only** vs **前向+反向** 的 cache:

| | fp32 | bf16 | bf16 省 |
|---|---|---|---|
| 前向 only cache | 0.220G | 0.176G | 0.04G(20%) |
| **前向+反向 cache** | **3.783G** | **2.422G** | **1.36G(36%)** |

**真相:bf16 的 cast 确实生效,前向中间 state 是 bf16(省 20%),反向图 cache 从 3.78G 降到 2.42G(省 36%,1.36G)。机制覆盖成立,报告选项 D 的内存机制推断方向正确。**

但这个 ~1.36G 的节省,在真实训练里被 **working_set 削顶机制**完全吞掉:
- fp32 的真实 cache 需求 > 削顶线 → 被压平到 8.97G
- bf16 把需求降低 ~1.36G,但**降低后的需求仍 > 削顶线** → 同样被压平到 8.96G
- 两版的 `active+cache` 都精确顶在 12.06-12.07G(95% working_set),看不出差异

**这正是 v2 报告 §2 核心发现的体现:cache 被 working_set 软上限削顶,真实需求被盖住,任何"降低需求"的优化在削顶区都看不到收益。** bf16 把需求从"假设的 ~13.3G"降到"假设的 ~11.9G",但削顶线在 12.08G,两版都越线,都被压平。

要看到 bf16 的真实收益,必须在 **≥32GB 机器**(working_set ≥ 25.4G,削顶线 ≥ 24.1G)上重测——那时两版需求都低于削顶线,bf16 的 1.36G 节省会显形。

## 5. 对 v2 报告 / Roadmap 的修正建议

1. **选项 D 内存预估修正**:v2 §6 选项 D 写"粗估可省约 30% 总 cache"——本实验实测**机制方向对(反向图 cache 省 36%),但在 16GB 削顶区无法兑现**。应补注:"省的量(~1.36G)小于削顶裕度,16GB 上看不到收益;需 ≥32GB 机器或与 cache_limit 叠加才显形。"

2. **选项 D 数值风险修正**:v2 §6 写"bf16 递归 200+步可能累积数值误差"——本实验实测**不发散**(误差前 30 步饱和在 ~1%)。数值风险这一栏可从 ⚠️ 降级。

3. **选项关系修正**:v2 §6 说"D 与 cache_limit 可叠加"。本实验证实:**bf16 + c4G 是 16GB 上唯一让 active+cache 脱离削顶的组合**(7.13G,56% working_set,compressor 0.92G 压力绿)。单独 bf16 不够,但 bf16+c4G 有效。

4. **state 距离判据修正**:补丁预判"state 重合 → 行为过"。实测反例:行为一致但 state 余弦仅 0.305。对 state tuning,**应以行为 A/B(loss + 解码)为准,state 距离只作辅助参考**。

5. **内存单位规矩**(已写进 AGENTS.md):全仓统一 GB(÷10⁹),禁止 GiB 混用。v2 报告的 12.71G(working_set)与脚本里的 11.84G 是同块内存的两种口径,曾导致"超过上限"的假象。

## 6. 给你的裁决建议(仅供参考,不替你定)

基于三笔账,D 的定位是:

- **作为"16GB 长样本内存救星"** → ❌ 不成立。削顶机制盖住了收益,单独 D 救不动。
- **作为"数值/质量安全的常规优化"** → ✅ 成立。loss 不退化、不发散、风格不崩。可与 cache_limit 叠加(bf16+c4G = 7.13G 绿档)。
- **作为"≥32GB 机器上的内存优化"** → ⚪ 待验证。机制上省 1.36G,大机器上应能显形,但本实验无 ≥32GB 数据。

可能的裁决方向(供参考):
- (a) **D 否决**:16GB 目标场景下收益不可见,不值改主路的代价。保留实验分支。
- (b) **D 有条件采纳**:不入默认路径,但作为"大内存模式"的可选开关(与 cache_limit 组合),给未来 ≥32GB 用户。
- (c) **D 转向 C+D 组合**:16GB 上主推 cache_limit(C 类),bf16 作为大机器可选叠加项。
