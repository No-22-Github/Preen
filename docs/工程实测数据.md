## 一、训练超参与曲线

产品当前默认学习率为 **0.0001**,最低学习率为 **0.00001**,走 warmup + cosine。
不是 RWKV-PEFT 老教程的 1.0:实测 lr=1.0 会让 state 数值爆炸,std 冲到正常值的
50~100 倍并退化成无条件偏置。下表历史曲线使用的是当时显式传入的 lr=0.01,
用于记录已完成实验,不代表当前产品默认值。

state std 目前只记录、不报警。旧的 ">1.0 预警线"已作废——官方 roleplay state 的
std 约 1.385 也工作正常,所以健康区间没有统一阈值。

实测 loss / state std 曲线:

| 模型 | 数据 | epoch | loss | state std |
|---|---|---|---|---|
| 0.4B | NekoQA 200 条 | 1 → 2 → 3 | 3.97 → 3.12 → 2.36 | 0.11 → 0.12 |
| 1.5B | NekoQA 200 条 | 1 → 2 | 2.48 → 1.71 | 0.12 → 0.12 |

### 早停行为(1.5B,充分训练)

1.5B 跑满(20 epoch 上限,patience=3)时的 held-out 曲线,是一个典型的过拟合样本:

```
ep0: train_loss=2.716  held_out=3.776  std=0.127  ★ BEST
ep1: train_loss=1.459  held_out=2.740  std=0.144  ★ BEST
ep2: train_loss=0.714  held_out=2.984  std=0.159
ep3: train_loss=0.382  held_out=3.233  std=0.174
ep4: train_loss=0.153  held_out=3.423  std=0.183   → 早停 @ ep4
```

train_loss 一路降到 0.15,但 held-out 从 ep1 之后持续回升。早停正确地停在 ep1
(held-out 最低)保存的 state 上。全程 state std 在 0.13~0.18,没有爆炸。
这也是"废除固定 epoch、改用 held-out 早停"的实际收益。

## 二、内存实测

### GPU 硬上限(本机 Apple M5 16GB,`mx.metal.device_info()` 真理值)

```
max_recommended_working_set_size = 12.71 G   总 working set 软上限
max_buffer_length                =  9.54 G   单个 Metal buffer 上限
memory_size                      = 16.00 G
```

削顶线 = 12.71 × 0.95 ≈ **12.07 G**。这是 16GB 机器上一切内存判定的基准线。

### cache 削顶机制(关键)

MLX 的 Metal allocator 回收的 buffer 会留在池里复用,不还给系统,所以 cache 单调只增不减。
当 `active + cache` 接近 working_set 软上限的 95% 时,MLX 停止扩大 cache 池——两个模型都精确顶在这条线上:

| 模型 | active | cache | active + cache | 占 working_set |
|---|---|---|---|---|
| 0.4B | 0.93G | 11.15G | 12.08G | 95% |
| 1.5B | 3.11G | 8.96G | 12.07G | 95% |

推论:被削顶时看到的 cache 稳态值(0.4B 11.15G、1.5B 8.96G)是削顶值,不是真实需求。
1.5B 的 cache 看起来比 0.4B "低",只是因为它的 active 更大(3.11 vs 0.93),挤占了 cache 空间,差值刚好对得上。

cache 的增长由**数据里最长样本的 token 数**驱动:逐样本追踪显示,cache 只在长样本出现时跳变,
短样本几乎不动;池子撑大后就永久保留。

### cache_limit 扫描(1.5B NekoQA 200,100 步同口径)

`--cache-limit-gb` 压的是这块结果缓存。扫描不同上限下的速度与压力:

| cache_limit | 稳态 cache | vm 压力 | ms/step | 判定 |
|---|---|---|---|---|
| 4G | 4.00G | 🟢 绿 | 1242 | 最快的绿档 |
| 6G | 6.00G | 🟡 黄 | 1117 | |
| 8G | 8.00G | 🟡 黄 | 1775 | |
| 无限(基准) | ≥8.97G(削顶) | 🟡 黄 | 2552 | |
| 10G | ~9.0G(中止) | 🔴 换页 | 6000~12000 | 系统重度换页,跑不动 |

相对无限档基准,c4G 快约 51%。**16GB 机器训练 1.5B,压力保持绿的最大 cache_limit 是 4G。**
产品 CLI 的 `--cache-limit-gb` 缺省 `auto`(物理内存 × 25%,16G 机约 4.3G)正是落在这一档。

> 注意:cache_limit 只管结果缓存,压不住前向计算图里的临时激活(fp32 wkv 循环的中间张量)。
> 后者只随 ctx 变化,ctx 过大导致超线时,只能降 ctx,调 cache_limit 没用。
> 可操作的 ctx_len 档位表见 [快速上手](快速上手.md#step-3--训练)。

### 峰值内存(ctx 512,bsz 1,bf16 权重 + fp32 state)

| 模型 | 权重 | 训练峰值 | 每步耗时 | state 大小 | 16GB |
|---|---|---|---|---|---|
| 0.4B | ~0.9G | 1.39G | 0.18s | 6.0MB(24 层 × 16 头) | 舒适 |
| 1.5B | ~3.0G | 4.00G | 0.29s | 12.6MB(24 层 × 32 头) | 舒适 |

> 这里的"训练峰值"是短样本、未削顶时的 MLX allocator 分配量;长样本会因 fp32 wkv 循环临时激活
> 顶到削顶线。两者不矛盾:前者是分配下限,后者是长样本压力上限。内存单位统一 GB(÷10⁹)。

### 内存汇报口径

`mx.get_peak_memory()` 严重漏报,只算 MLX allocator,不含 Metal wired memory。汇报内存时给这几个数:

1. `mx.get_active_memory()` — 当前活跃计算(与 cache 互斥)
2. `mx.get_cache_memory()` — 回收池(对账活动监视器的主要口径)
3. `active + cache` — 总占用,对比 working_set 上限,>95% 即削顶
4. `vm_stat` compressor — 内存压力源(<2G 🟢 / 2~4G 🟡 / >4G 🔴)

测量要跑到 ≥75 步才到稳态或削顶。精度方案(bf16 权重 + fp32 state)与 w 分布的详细分析见
[decision-precision.md](decision-precision.md)。

## 三、`.pth` 导出方向裁决

x070(RWKV-7)导出 state 时**存原样,不做 transpose**。这是经 RWKV-Runner 源码 +
Windows 真机双重确认的最终结论。

依据在 RWKV-Runner 的 `backend-python/utils/rwkv.py`(`load_rwkv_state`,约 line 843):

```python
if model.model.version >= 7:                     # x070
    state_tuned[i*3+1] = time_state.to(float)    # 原样,不 transpose
else:                                             # v5/v6
    state_tuned[i*3+1] = time_state.transpose(1, 2)  # 才转置
```

Runner 自带的 model.py 把 x070 判为 version=7,走上面那条不转置的分支。v5/v6 的
`transpose(1,2)` 是为了把 fla 训练方向转成 CUDA 方向,x070 不需要。

导出格式:

| 项 | 值 |
|---|---|
| 容器 | torch 的 zip+pickle `.pth`(由 `pth_io.write_pth` 纯 Python 生成,无 torch) |
| 键名 | `blocks.{i}.att.time_state` |
| 形状 | `(H, D, D)` = `(num_heads, head_dim, head_dim)` |
| dtype | fp32(Runner 加载后转 float) |
| 转置 | x070 存原样;v5/v6 兼容路径存 swapaxes |

导出器带 round-trip 验证(写回 → 读回 → 逐张量比对,max diff 0.0)。读写 `.pth` 均由
`src/statetuner/pth_io.py` 纯 Python 完成——`read_pth` 与 `torch.load` 逐字节等价(3 个真实
模型 798/798 张量验证),`write_pth` 产物 `torch.load(weights_only=True)` 可读、RWKV Runner
可挂载。挂载步骤见 [Runner 挂载验收](Runner挂载验收.md)。

> 一条教训:Mac 侧的闭环验证(round-trip、挂载等价性)只能证明"内部自洽",证明不了
> "和外部消费者兼容"。导出格式最终以 Windows 真机验收为准——早期正是因为只信 Mac 侧闭环,
> 一度把方向判反了。

## 四、CLI 架构与约定

`cli.py` 是薄壳,业务编排下沉在 `service` / `inference` / `inspection` / `events` 四层。
目标是删掉 `cli.py` 后,SwiftUI sidecar 仅依赖这四层就能实现全部同等功能(终端 REPL 除外)。

- **eval**:请求/结果结构、校验、生成循环在 `service`,engine 由调用方注入(便于用 FakeEngine 单测)。
- **chat**:state 校验(层数/shape/rwkv7 兼容)在 `inspection.validate_state_for_model`,
  初始加载和运行中 `/state` 切换共用。REPL 循环与流式回调留在 CLI。
- **preview**:摘要行 `[stop=…, tokens=…, t/s]` 由 `GenerationResult.summary_line()` 单一来源产出。

退出码全命令一致:**0** 成功 / **2** 输入错误 / **1** 运行失败 / **130** 中断。
stdout / stderr 分离:JSON 与结果文本走 stdout,`#` 状态、进度、错误走 stderr。

`--template` 支持范围(有意设计):

| 命令 | 支持模板 | 默认 |
|---|---|---|
| train | nekoqa | nekoqa |
| eval | nekoqa / g1g | — |
| chat | raw / nekoqa / g1g | g1g |
| preview | raw / nekoqa / g1g | raw |

事件流(`events.py`)是 JSON lines,SwiftUI sidecar 往后可直接消费它驱动进度面板与 loss 曲线,无需另设协议。
