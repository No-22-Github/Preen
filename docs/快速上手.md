# 快速上手 — 用 NekoQA 跑通第一次微调

> 本文是一份**自包含的分步教程**:从零开始,用仓库内置的 NekoQA 数据集,
> 训练一个能改变 RWKV 模型说话风格的初始状态(state),并导出可在
> RWKV Runner 中挂载的 `.pth` 文件。所有命令可直接复制粘贴。
>
> 概览与全部子命令见 [README](../README.md);原理与背景见
> [P0 理论指南](P0-理论指南.md)。

---

## 你会得到什么

训练完成后,你会拿到一个 `state.npz`(以及导出的 `state.pth`)。
把它注入 RWKV 模型作为每层的初始状态 S₀,模型从此状态启动时,
就会从"普通助手"变成"NekoQA 猫娘风格"——开口带"喵/主人"、
回答里夹括号动作描写。

**这是 style(风格)注入,不是 fact(事实)注入。**
state tuning 学的是说话方式,不会让模型学会新的知识或翻译内容。
详见文末 FAQ。

## 前置要求

- **Apple Silicon Mac**(M1+,本项目在 M5 / 16GB 上验证)
- **[uv](https://docs.astral.sh/uv/)** 包管理器
- Python 3.11(uv 会自动管理,无需手动装)
- 仓库已 clone,并在仓库根目录执行 `uv sync`(安装依赖)

```bash
git clone <repo-url> Preen && cd Preen
uv sync                    # 安装依赖(mlx-lm + torch + typer 等)
```

> 16GB 机器默认训练时,RSS 会到 11~12GB,期间其他重型 App 建议关掉。
> 内存吃紧时可调 `--cache-limit-gb`(见 Step 3 与 FAQ)。

## Step 1 — 获取模型权重

本项目不包含大模型文件(被 gitignore)。需下载原始 RWKV-7 权重到 `models/`:

| 文件 | 用途 | 大小 | 来源 |
|---|---|---|---|
| `rwkv7-g1d-0.4b-20260210-ctx8192.pth` | 0.4B 原始权重(待转换) | 902M | [BlinkDL/rwkv7-g1](https://huggingface.co/BlinkDL/rwkv7-g1/tree/main) |

> 1.5B 同系列权重可选(`rwkv7-g1g-1.5b-*.pth` 等),训练管线相同,
> 但 16GB 机器内存更紧,首次跑建议用 0.4B。

**转换所需的校验 fixture(`tools/fixtures/rwkv7_hf_template.json`)和
tokenizer 文件(`assets/rwkv_world_tokenizer/`)已 vendor 进仓库,
转换过程零外部下载。**

## Step 2 — 转换模型

RWKV 官方发布的是原生 `.pth`,MLX 需要的是 fla HF(safetensors)格式。
用仓库内置的独立转换器:

```bash
uv run python tools/convert_rwkv7_to_hf.py \
    --rwkv7 models/rwkv7-g1d-0.4b-20260210-ctx8192.pth \
    --output models/converted/rwkv7-g1d-0.4b --precision bf16
```

转换器**不依赖 fla / triton**(macOS 无 triton wheel)。键名映射规则、
维度校验、tokenizer 复制都内置;`--reference` / `--tokenizer-src`
两个参数缺省走仓库内置资源,**只有在上游 schema 漂移时才需要显式提供**
(详见转换器文件头 docstring)。

产物:`models/converted/rwkv7-g1d-0.4b/` 下会有
`model.safetensors` + `config.json` + tokenizer 5 文件。

## Step 3 — 训练

```bash
uv run statetuner train \
    --model models/converted/rwkv7-g1d-0.4b \
    --data train_data/NekoQA_10k/nekoqa_smoke_200.json \
    --template nekoqa \
    --out state.npz \
    --events-file events.jsonl \
    --lr 0.01 --epochs 3 --ctx-len 512 --no-early-stop --seed 42 \
    --export-pth --pth-out state.pth
```

### 参数说明

| 参数 | 含义 |
|---|---|
| `--model` | Step 2 转换后的 HF 模型目录 |
| `--data` | 训练数据。`nekoqa_smoke_200.json` 是 NekoQA 10k 里抽的 200 条快测集;全量是 `NekoQA-10K.json` |
| `--template` | **任务模板。训练与推理必须用同一个模板**,否则 token 编码不同构,state 无效。当前训练只支持 `nekoqa` |
| `--out` | 训练产物 `.npz`(Preen 内部格式) |
| `--events-file` | 训练事件流(JSON lines,loss/std/lr/epoch)同时写文件,方便事后看曲线 |
| `--lr 0.01` | **学习率默认 0.01,不是 RWKV-PEFT 老教程的 1.0**——1.0 会让 state 数值爆炸(std 飙到正常值 50~100 倍) |
| `--epochs 3` | NekoQA 200 条 3 个 epoch 已能见到明显风格注入 |
| `--ctx-len 512` | 上下文长度。200 条 smoke 集基本不会触发截断 |
| `--no-early-stop` | 关掉 held-out 早停(小数据集上想跑满看曲线时用;正式训练建议保留早停) |
| `--seed 42` | 固定种子,结果可复现 |
| `--export-pth` / `--pth-out` | 训完顺手导出 RWKV Runner 可挂载的 `.pth`(也可单独用 `export` 命令,见 Step 7) |

### 训练精度

当前训练是 **bf16 权重 + fp32 state**(main 分支唯一实现,不可切换):

- **权重**:Step 2 用 `--precision bf16` 转换,训练全程冻结、不更新
- **state(可训练参数)**:`make_state_params(dtype=mx.float32)`,即每层 64×64 的 S₀ 全程 fp32
- **wkv 循环**:bf16 权重 × fp32 state,按 MLX 类型提升规则(bf16+fp32=fp32)整个循环被提升成 fp32

> **为什么不用 bf16 state(所谓 "D 方案")**:实测有劣化——配对判据 4 红 1 绿,
> `15b_s42` 有确凿退化单例(Q8 连续输出"啊"121 字)。该方案已废弃,代码已删,
> 详见 [decision-precision.md](decision-precision.md)。
> 这也是 0.4B 训练 RSS 飙到 11~12GB 的机制原因(fp32 循环比 bf16 高 ~1.7G),
> 见下面的内存说明。

### 训练时长与内存

- **0.4B × 200 条 × 3 epoch ≈ 8 分钟**(单条命令内可完成)
- 1.5B 同条件时长更长,且 16GB 机器内存更紧(见下方 ctx 上限表)
- **`--cache-limit-gb` 默认 `auto`** = 物理内存 × 25%(16GB 机 ≈ 4.3G)。
  它压住的是 MLX 结果缓存(cache),**压不住**前向计算图里的临时激活
  (fp32 wkv 循环的中间张量,这部分随 ctx 线性增长,见下)。

### ctx_len 上限(1.5B + fp32 state,M5 / 16GB,auto cache 4.3G)

> 实测于 G1H 1.5B,fps32 state,3 档 20 步探针。判据用 AGENTS.md 的
> **ms/step 拐点法**——GPU_peak 是否超 working_set 削顶线(12.71G)是硬标准,
> ms/step 是否随 step 飙升是软标准(内存压力开始拖慢计算)。

| `--ctx-len` | GPU_peak | ms/step 趋势 | 判定 |
|---|---|---|---|
| **256** | 7.5G | 平稳(2.4→2.9s) | ✅ **安全档** |
| 320 | 8.8G | 略涨(4.0→5.7s) | ⚠️ 压力区边缘 |
| 384 | 10.1G | 明显飙升(5.3→10.3s) | ❌ 已在降速区 |
| 512 | 12.8G | 换页/卡顿 | ❌ 超 12.71G 削顶线 |

**结论:16GB 机器训练 1.5B,`--ctx-len` 建议设 256(安全),最高别超 320。**
默认值 512 在 1.5B + fp32 下会超线触发换页。0.4B 内存余量更大,512 安全。

> 为什么 512 不安全?单步内存构成:权重 3.05G + state 0.02G + cache 4.29G
> + **fp32 wkv 循环临时 ~5.4G** = 峰值 12.8G,超 Metal working_set 削顶线(12.71G)。
> cache_limit 只管结果缓存,管不了这块临时激活——它只随 ctx 变化,每 64 token
> 涨约 1.25G。降 `--cache-limit-gb` 没用,只能降 `--ctx-len`。

> ctx_len 不是越大越好:NekoQA smoke 200 条中位数 103 token,全量 10k 中位数 313。
> smoke 数据设 ctx 256 足够(基本不截断);全量长答案多,ctx 256 会截断约 30%
> 样本,但截断只丢答案尾部,state tuning 学风格不受实质影响。

### 预期结果

0.4B × 200 条 × 3 epoch 的实测曲线(loss / state std):

| epoch | loss | state std |
|---|---|---|
| 1 | 3.97 | 0.11 |
| 2 | 3.12 | 0.12 |
| 3 | 2.36 | 0.12 |

训练事件实时输出到 stdout(JSON lines),同时写入 `events.jsonl`。
看 loss 曲线:`grep '"type": "epoch_end"' events.jsonl`。

## Step 4 — 预览:对比有无 state

```bash
uv run statetuner preview \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz \
    --prompt "你好呀，今天想做什么？" --template nekoqa --ab
```

`--ab` 会**双路输出**:有 state(tuned)vs 无 state(基线),一眼看出
风格注入的效果——基线是普通助手腔,tuned 变成猫娘腔(括号动作 + 喵/主人)。

```bash
# 轻采样预览(temperature=0 仍是可复现贪心)
uv run statetuner preview \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --prompt "你好" --template nekoqa \
    --temperature 0.8 --top-p 0.9 --seed 42 --json
```

> **模板一致性铁律**:`--template nekoqa` 必须与训练时相同。
> 模板驱动 prompt 包装与 stop sequences,训练/推理不同模板会让 state 失效。

## Step 5 — 评估:在 held-out 上生成

```bash
uv run statetuner eval \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --template nekoqa \
    --data train_data/NekoQA_10k/nekoqa_smoke_200.json \
    --limit 5
```

对数据集逐条生成(state 已注入),输出每条的 `OUT`(模型生成)与
`REF`(参考答案)。NekoQA 推理除了 EOS,还会在模型开始生成下一轮
`User:` 角色标记时停止并剥离该标记,避免自问自答。

## Step 6 — 交互聊天

```bash
# 模型常驻交互;运行中可用 /state 动态切换 state
uv run statetuner chat \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --template nekoqa \
    --max-tokens 200 --temperature 0.6 --top-p 0.7
```

> **`chat` 默认模板是 `g1g`(RWKV7-G1 原生对话)。**
> 用本教程训出来的猫娘 state 时,**必须显式 `--template nekoqa`**——
> state 是按 nekoqa 模板训的,换模板会失效。

`chat` 内置命令(运行中输入):

| 命令 | 作用 |
|---|---|
| `/state PATH` | 动态加载新 state,下一轮立即生效 |
| `/state off` | 切回零 state 基线 |
| `/ab on` | 同时输出 tuned state / 基线 |
| `/config` | 查看当前采样配置 |
| `/help` | 查看全部命令 |
| `/quit` | 退出 |

`chat` 默认**流式输出**;一次性输出加 `--no-stream`。
单轮 `preview` 可显式加 `--stream`,但流式不与 `--ab` / `--json` 混用。

## Step 7 — 导出与挂载

Step 3 用 `--export-pth` 已一步导出 `state.pth`。也可单独导出:

```bash
uv run statetuner export --state state.npz --out state.pth
```

导出器会做 round-trip 验证(写回 → 读出 → 逐张量比对)。键名
`blocks.{i}.att.time_state`,形状 `(H,D,D)` fp32,**x070(RWKV-7)原样
不转置**。

导出的 `.pth` 可直接在 **RWKV Runner(Windows)** 中作为模型初始 state
加载——Runner 检测到 `blocks.{i}.att.time_state` 键后自动启用
tuned-state 路径。挂载步骤见
[Runner挂载验收.md](Runner挂载验收.md)。

## 常见问题(FAQ)

**Q: state tuning 学会了什么?能用来翻译/注入知识吗?**
A: 实测只学**风格**(说话方式、人设、输出格式),不学**事实**。
内容映射(如翻译)实测不可用。所以任务设计要按"风格注入"边界来。

**Q: `--ctx-len` 该怎么选?16GB 机器能开多大?**
A: 看「ctx_len 上限」表。**16GB 机器训练 1.5B,ctx 设 256 安全,最高 320。**
0.4B 内存余量大,512 安全。超限不会立即崩(MLX 会换页硬扛),但 ms/step
翻倍、训练变极慢。关键认知:这里的大头是 **fp32 wkv 循环的临时激活**,
它只随 ctx 变化,降 `--cache-limit-gb` 压不住它——只能降 ctx。

**Q: `--cache-limit-gb` 该怎么调?能解决 OOM 吗?**
A: 默认 `auto`(物理内存×25%,16G 机≈4.3G)。它压的是 **MLX 结果缓存**
(可复用的前向结果),压不到前向计算图里的临时激活。所以 ctx 过大导致
超线时,调小 `--cache-limit-gb` **没用**——得降 ctx。它主要在多任务争内存
时有用(把 MLX 占的 cache 让出来给系统)。

**Q: 训练 loss 曲线怎么看?历史数据可比吗?**
A: `grep '"type": "epoch_end"' events.jsonl` 看 per-epoch loss/std。
注意:`--cache-limit-gb` 的 auto 默认改变了训练口径,历史训练曲线
(在没设 cache_limit 下跑的)与现在不再严格可比。

**Q: 16GB 机器训练会爆内存吗?**
A: 0.4B 在 ctx 512 下安全;**1.5B 在 ctx 512 会超 Metal working_set 削顶线
(12.71G),实测峰值 12.8G,触发换页**。1.5B 请用 ctx 256。内存机制与
红线标定详见 [decision-precision.md](decision-precision.md)。

**Q: 训练和推理用不同 `--template` 会怎样?**
A: state 失效。模板驱动 prompt 包装 + stop sequences + 编码路径,
训练/推理必须同模板,才能保证 token 编码同构。
g1g 模板的格式敏感性见 [g1g-decode-alignment.md](g1g-decode-alignment.md)。

**Q: 为什么 lr 是 0.01 不是 1.0?**
A: 实测 lr=1.0 会让 state std 飙到正常值 50~100 倍,变成无条件偏置。
lr=0.01 让 state 温和生长,保留对输入的条件响应。

**Q: 0.4B 输出有重复是怎么回事?**
A: 0.4B 基座固有缺陷,有/无 state 都会重复,不是 state tuning 引入的。
