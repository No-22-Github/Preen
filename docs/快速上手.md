# 快速上手 — 用 NekoQA 跑通第一次微调

> 一份自包含的分步教程:从零开始,用仓库内置的 NekoQA 数据集,训练一个能改变
> RWKV 说话风格的初始状态(state),再导出成可在 RWKV Runner 中挂载的 `.pth`。
> 所有命令都能直接复制粘贴。
>
> 这里走的是命令行。图形界面(SwiftUI)还在路上,`statetuner` CLI 是当前完整可用的入口,
> 也是往后 UI 背后实际执行的引擎——所以这套流程不会白学。
>
> 概览与全部子命令见 [README](../README.md);原理与背景见
> [P0 理论指南](P0-理论指南.md)。

---

## 你会得到什么

训练完成后,你手上会有一个 `state.npz`,以及导出的 `state.pth`。把它作为每层的初始状态 S₀
注入 RWKV 模型,模型从这个状态启动时,就从"普通助手"变成了"NekoQA 猫娘风格"——
开口带"喵/主人",回答里夹着括号动作描写。

这里注入的是**风格**,不是**事实**。state tuning 学的是说话方式,不会让模型学到新知识
或记住具体内容。这条边界很重要,细节见文末 FAQ。

## 前置要求

- Apple Silicon Mac(M1 及以上,本项目在 M5 / 16GB 上验证)
- [uv](https://docs.astral.sh/uv/) 包管理器
- Python 3.11,uv 会自动管理,不用手动装
- 仓库已 clone,并在根目录跑过 `uv sync`

```bash
git clone <repo-url> Preen && cd Preen
uv sync                    # 安装依赖(mlx-lm + torch + typer 等)
```

> 16GB 机器按默认参数训练时,RSS 会到 11~12GB,这期间最好把其他重型 App 关掉。
> 内存吃紧可以调 `--cache-limit-gb`,见 Step 3 和 FAQ。

## Step 1 — 获取模型权重

本项目不含大模型文件(已 gitignore)。需要先把原始 RWKV-7 权重下到 `models/`:

| 文件 | 用途 | 大小 | 来源 |
|---|---|---|---|
| `rwkv7-g1d-0.4b-20260210-ctx8192.pth` | 0.4B 原始权重(待转换) | 902M | [BlinkDL/rwkv7-g1](https://huggingface.co/BlinkDL/rwkv7-g1/tree/main) |

> 也可以用 1.5B 的同系列权重(`rwkv7-g1g-1.5b-*.pth` 等),训练管线完全一样,
> 但 16GB 机器上内存更紧,第一次跑建议先用 0.4B。

转换需要的校验 fixture(`tools/fixtures/rwkv7_hf_template.json`)和 tokenizer 文件
(`assets/rwkv_world_tokenizer/`)都已 vendor 进仓库,转换过程零外部下载。

---

## Step 2 — 转换模型

RWKV 官方发布的是原生 `.pth`,MLX 这边要的是 fla HF(safetensors)格式。
用仓库自带的独立转换器过一道:

```bash
uv run python tools/convert_rwkv7_to_hf.py \
    --rwkv7 models/rwkv7-g1d-0.4b-20260210-ctx8192.pth \
    --output models/converted/rwkv7-g1d-0.4b --precision bf16
```

转换器不依赖 fla 和 triton(macOS 没有 triton wheel)。键名映射规则、维度校验、
tokenizer 复制全都内置;`--reference` 和 `--tokenizer-src` 两个参数缺省走仓库内置资源,
只有在上游 schema 漂移时才需要显式指定,细节见转换器文件头的 docstring。

产物在 `models/converted/rwkv7-g1d-0.4b/` 下:`model.safetensors` + `config.json` + tokenizer 5 文件。

## Step 3 — 训练

```bash
uv run statetuner train \
    --model models/converted/rwkv7-g1d-0.4b \
    --data train_data/NekoQA_10k/nekoqa_smoke_200.json \
    --template nekoqa \
    --out state.npz \
    --events-file events.jsonl \
    --lr 0.01 --epochs 3 --ctx-len 512 --no-early-stop --seed 42 \
    --export-pth --pth-out state.pth
```

### 参数说明

| 参数 | 含义 |
|---|---|
| `--model` | Step 2 转换后的 HF 模型目录 |
| `--data` | 训练数据。`nekoqa_smoke_200.json` 是从 NekoQA 10k 里抽的 200 条快测集;全量是 `NekoQA-10K.json` |
| `--template` | 任务模板。训练和推理必须用同一个,否则 token 编码不同构、state 失效。当前训练只支持 `nekoqa` |
| `--out` | 训练产物 `.npz`(Preen 内部格式) |
| `--events-file` | 训练事件流(JSON lines,含 loss/std/lr/epoch)同时写到文件,方便事后看曲线 |
| `--lr 0.01` | 学习率默认 0.01,不是 RWKV-PEFT 老教程的 1.0——1.0 会让 state 数值爆炸,std 飙到正常值的 50~100 倍 |
| `--epochs 3` | 200 条数据跑 3 个 epoch,已经能看到明显的风格注入 |
| `--ctx-len 512` | 上下文长度。200 条 smoke 集基本不会触发截断 |
| `--no-early-stop` | 关掉 held-out 早停。想在小数据集上跑满、看完整曲线时用;正式训练建议保留早停 |
| `--seed 42` | 固定随机种子,结果可复现 |
| `--export-pth` / `--pth-out` | 训完顺手导出可挂载的 `.pth`;也可以单独用 `export` 命令,见 Step 7 |

### 训练精度

当前训练是 bf16 权重 + fp32 state,这是 main 分支的唯一实现,不可切换:

- **权重**:Step 2 用 `--precision bf16` 转换,训练全程冻结、不更新
- **state(可训练参数)**:`make_state_params(dtype=mx.float32)`,每层 64×64 的 S₀ 全程 fp32
- **wkv 循环**:bf16 权重乘 fp32 state,按 MLX 的类型提升规则(bf16+fp32=fp32),整个循环被提升成 fp32

> 为什么不用 bf16 state(即所谓 "D 方案"):实测有劣化——配对判据 4 红 1 绿,
> `15b_s42` 出现确凿的退化单例(Q8 连续输出"啊"121 字)。该方案已废弃、代码已删,
> 裁决过程见 [decision-precision.md](decision-precision.md)。
> 这也是 0.4B 训练 RSS 会飙到 11~12GB 的机制原因:fp32 循环比 bf16 高约 1.7G,
> 详见下面的内存说明。

### 训练时长与内存

- 0.4B × 200 条 × 3 epoch 大约 8 分钟,单条命令内能跑完
- 1.5B 同条件下更久,而且 16GB 机器内存更紧,见下方 ctx 上限表
- `--cache-limit-gb` 默认 `auto`,取物理内存的 25%(16GB 机约 4.3G)。它压的是
  MLX 结果缓存,压不住前向计算图里的临时激活——也就是 fp32 wkv 循环的中间张量,
  这部分随 ctx 线性增长,见下。

### ctx_len 上限(1.5B + fp32 state,M5 / 16GB,auto cache 4.3G)

> 实测于 G1H 1.5B、fp32 state,分 3 档做了 20 步探针。判据沿用 AGENTS.md 的
> ms/step 拐点法:GPU_peak 是否超过 working_set 削顶线(12.71G)是硬标准,
> ms/step 是否随 step 飙升是软标准(内存压力开始拖慢计算)。

| `--ctx-len` | GPU_peak | ms/step 趋势 | 判定 |
|---|---|---|---|
| **256** | 7.5G | 平稳(2.4→2.9s) | ✅ 安全档 |
| 320 | 8.8G | 略涨(4.0→5.7s) | ⚠️ 压力区边缘 |
| 384 | 10.1G | 明显飙升(5.3→10.3s) | ❌ 已在降速区 |
| 512 | 12.8G | 换页/卡顿 | ❌ 超 12.71G 削顶线 |

结论:16GB 机器训练 1.5B,`--ctx-len` 建议设 256,最高别超 320。默认值 512 在
1.5B + fp32 下会超线、触发换页。0.4B 内存余量大得多,512 是安全的。

> 512 为什么不安全?单步内存拆开看:权重 3.05G + state 0.02G + cache 4.29G,再加上
> fp32 wkv 循环的临时张量约 5.4G,峰值就到了 12.8G,越过 Metal working_set 削顶线(12.71G)。
> cache_limit 只管结果缓存,管不到这块临时激活——而它只随 ctx 变化,每 64 token 涨约 1.25G。
> 所以调小 `--cache-limit-gb` 没用,只能降 `--ctx-len`。

> ctx_len 也不是越大越好。NekoQA smoke 200 条中位数 103 token,全量 10k 中位数 313。
> smoke 集设 ctx 256 就够(基本不截断);全量长答案多,ctx 256 会截断约 30% 的样本,
> 但截断只丢答案尾部,对 state tuning 学风格没有实质影响。

### 预期结果

0.4B × 200 条 × 3 epoch 的实测曲线(loss / state std):

| epoch | loss | state std |
|---|---|---|
| 1 | 3.97 | 0.11 |
| 2 | 3.12 | 0.12 |
| 3 | 2.36 | 0.12 |

训练事件实时输出到 stdout(JSON lines),同时写入 `events.jsonl`。
看 loss 曲线:`grep '"type": "epoch_end"' events.jsonl`。

## Step 4 — 预览:对比有无 state

```bash
uv run statetuner preview \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz \
    --prompt "你好呀，今天想做什么？" --template nekoqa --ab
```

`--ab` 会**双路输出**:有 state(tuned)对比无 state(基线),风格注入的效果一眼就能看出来——
基线是普通助手腔,tuned 变成猫娘腔(括号动作 + 喵/主人)。

```bash
# 轻采样预览(temperature=0 仍是可复现的贪心解码)
uv run statetuner preview \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --prompt "你好" --template nekoqa \
    --temperature 0.8 --top-p 0.9 --seed 42 --json
```

> 模板必须一致:`--template nekoqa` 要和训练时相同。模板决定了 prompt 包装和 stop
> sequences,训练/推理用不同模板会让 state 失效。

## Step 5 — 评估:在 held-out 上生成

```bash
uv run statetuner eval \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --template nekoqa \
    --data train_data/NekoQA_10k/nekoqa_smoke_200.json \
    --limit 5
```

对数据集逐条生成(state 已注入),每条输出模型生成的 `OUT` 和参考答案 `REF`。
NekoQA 推理除了 EOS,还会在模型开始生成下一轮 `User:` 角色标记时停下,并把该标记剥掉,
避免自问自答。

## Step 6 — 交互聊天

```bash
# 模型常驻交互;运行中可用 /state 动态切换 state
uv run statetuner chat \
    --model models/converted/rwkv7-g1d-0.4b \
    --state state.npz --template nekoqa \
    --max-tokens 200 --temperature 0.6 --top-p 0.7
```

> `chat` 的默认模板是 `g1g`(RWKV7-G1 原生对话)。用本教程训出来的猫娘 state 时,
> 一定要显式加 `--template nekoqa`——state 是按 nekoqa 模板训的,换模板就失效。

`chat` 运行中可以输入这些命令:

| 命令 | 作用 |
|---|---|
| `/state PATH` | 动态加载新 state,下一轮立即生效 |
| `/state off` | 切回零 state 基线 |
| `/ab on` | 同时输出 tuned state 和基线 |
| `/config` | 查看当前采样配置 |
| `/help` | 查看全部命令 |
| `/quit` | 退出 |

`chat` 默认流式输出;想一次性输出加 `--no-stream`。单轮 `preview` 可以显式加 `--stream`,
但流式不与 `--ab` / `--json` 混用。

## Step 7 — 导出与挂载

Step 3 加了 `--export-pth` 的话,`state.pth` 已经导好了。也可以单独导出:

```bash
uv run statetuner export --state state.npz --out state.pth
```

导出器会做 round-trip 验证:写回、读出、逐张量比对。键名是 `blocks.{i}.att.time_state`,
形状 `(H,D,D)`、fp32,x070(RWKV-7)原样不转置。

导出的 `.pth` 可以直接在 RWKV Runner(Windows)里作为模型初始 state 加载——Runner 检测到
`blocks.{i}.att.time_state` 键后会自动走 tuned-state 路径。挂载步骤见
[Runner挂载验收.md](Runner挂载验收.md)。

## 常见问题(FAQ)

**Q: state tuning 到底学会了什么?能拿来翻译或注入知识吗?**
A: 实测只学风格(说话方式、人设、输出格式),不学事实。内容映射(比如翻译)实测不可用。
所以任务设计要卡在"风格注入"这条边界内。

**Q: `--ctx-len` 怎么选?16GB 机器能开多大?**
A: 看上面的 ctx_len 上限表。16GB 训练 1.5B,ctx 256 安全,最高 320;0.4B 内存余量大,512 安全。
超限不会立即崩(MLX 会换页硬扛),但 ms/step 翻倍、训练变得极慢。要记住:这里的内存大头是
fp32 wkv 循环的临时激活,它只随 ctx 变化,调 `--cache-limit-gb` 压不住,只能降 ctx。

**Q: `--cache-limit-gb` 怎么调?能解决 OOM 吗?**
A: 默认 `auto`(物理内存的 25%,16G 机约 4.3G)。它压的是 MLX 结果缓存(可复用的前向结果),
碰不到计算图里的临时激活。所以 ctx 过大导致超线时,调小它没用,得降 ctx。它真正有用的场景是
多任务争内存时,把 MLX 占的 cache 让出来给系统。

**Q: 训练 loss 曲线怎么看?历史数据能对比吗?**
A: `grep '"type": "epoch_end"' events.jsonl` 看 per-epoch 的 loss/std。注意 `--cache-limit-gb`
的 auto 默认改变了训练口径,过去在没设 cache_limit 时跑的曲线,和现在的不再严格可比。

**Q: 16GB 机器训练会爆内存吗?**
A: 0.4B 在 ctx 512 下安全;1.5B 在 ctx 512 会越过 Metal working_set 削顶线(12.71G),
实测峰值 12.8G,触发换页,所以 1.5B 请用 ctx 256。内存机制和红线标定见
[decision-precision.md](decision-precision.md)。

**Q: 训练和推理用了不同的 `--template` 会怎样?**
A: state 失效。模板决定 prompt 包装、stop sequences 和编码路径,训练/推理必须同模板,
才能保证 token 编码同构。g1g 模板的格式敏感性见 [g1g-decode-alignment.md](g1g-decode-alignment.md)。

**Q: lr 为什么是 0.01,不是 1.0?**
A: 实测 lr=1.0 会让 state std 飙到正常值的 50~100 倍,退化成无条件偏置。
lr=0.01 让 state 温和生长,保留对输入的条件响应。

**Q: 0.4B 输出有重复是怎么回事?**
A: 这是 0.4B 基座本身的缺陷,有没有 state 都会重复,不是 state tuning 引入的。
