# P1 已知问题：ep04 state 训练质量不足

> 状态：未解决 / 待训练侧跟进　｜ 记录日期：2026-07-10
> 前置：导出方向问题已修复（x070 noswap），本问题与之无关。

## 现象

导出方向修正（noswap）后，Windows RWKV Runner 真机验收：
- **无 state 基线**：中文输入 → 通顺中文回复（基模型健康）
- **挂载 ep04 state**：中文输入 → 无拼写错误的英文，关键词命中，但**非翻译水准**（英文单词硬凑，语义不完整）

即：导出链路已完全正确（方向、格式、Runner 路径全部健康），剩余问题在训练出的 state 数值本身。

## 证据

**1. P0 golden 快照显示 ep04 当时就已退化（非导出引入）**

`tests/golden/translate_test.json` 中 ep04 注入后的实际输出 vs 参考答案：

| 输入 | 参考答案 | ep04 实际输出 |
|---|---|---|
| 今天下午三点开会… | The meeting is at 3 PM this afternoon… | Please remember to open the meeting to visit next week… |
| 由于连续降雨… | Due to continuous rainfall… | Due to the rains, the rains are light… |

即使训练集内的句子（模型见过的），输出也非精确翻译。P0 验收标准过松——"输出了英文"即判通过，未做语义校验。

**2. state 数值量级偏小**

| 来源 | 各层 std 均值 |
|---|---|
| 官方 roleplay state（3B，Runner 正常工作） | 1.385 |
| 我们 ep04（0.4B） | 0.152 |

差约 10×。疑似训练信号不足（注：不同任务/模型量级可能有差异，但 10× 差距值得怀疑）。

## 可能原因（均未定位，需后续实验）

1. **模型容量限制**：0.4B 模型可能本身翻译能力有限，state tuning 只调初始 state 榨不出足够翻译能力
2. **训练数据太少**：100 句训练集，可能不足以让 state 固化稳定的"中→英翻译"模式
3. **lr 配方问题**：P0 测出 lr=1.0 爆炸、lr=0.01 温和（std 0.15）。但官方正常 state std~1.4，可能需要找介于两者之间的 lr 区间
4. **state 初始化方式**：我们从全零初始化训练，而官方 roleplay state 疑似从真实语料 forward 累积而来——起点不同可能导致最终信号差异

## 不属于本问题的部分（已解决）

- ❌ 导出方向（swapaxes）→ 已修复为 x070 noswap，Windows 真机确认
- ❌ 导出格式（键名/shape/dtype）→ 与官方 roleplay state 逐项一致
- ❌ Runner 加载路径 → 官方 state 同链路正常，证明 Runner 健康

## 后续建议（供训练侧参考）

- 用更大模型（1.5B/2.9B）验证是否纯容量问题
- lr 扫描：0.01 / 0.05 / 0.1 / 0.3，观察 state std 与翻译质量的关系
- 尝试从真实中文语料 forward 累积的 state 作为训练起点，而非全零
- 引入语义级评估指标（非"输出了英文"即通过），如 BLEU 或人工语义打分
