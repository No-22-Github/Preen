## 二、`.pth` 导出器(任务①,核心暗坑)

> **⚠️ 修正记录(2026-07-10)**:本节初版基于 rwkv pip 0.8.32 的 v5/v6 transpose 逻辑,
> 误判导出方向为"需 swapaxes"。后经 Windows 真机验收 + RWKV-Runner 源码推翻:
> **x070(v7) 在 RWKV-Runner 中加载 state 不 transpose,导出器应存原样(noswap)**。
> 下方已更新为正确结论。完整过程见 [P1-任务①收尾报告.md](P1-任务①收尾报告.md)。

### 2.1 问题:转置方向

P0 只在 MLX 内部闭环(state 存成 npz,自己读回来用)。P1 要导出 `.pth` 给 RWKV Runner
等外部消费者挂载,这里有一个**版本相关的转置约定**。

三方实现的 state 递归方向(经源码+数值验证,可信):

| 实现 | 递归更新项 | 输出收缩 | 方向 |
|---|---|---|---|
| MLX `_wkv7_step_ops`(我们训练用) | `S[i,j] += v[i]·k[j]` | `y = S @ r`(r 缩并末维) | **方向 A** |
| BlinkDL CUDA kernel(rwkv7.cu,RWKV Runner 用) | `S[i,j] += v[i]·k[j]` | `y = S @ r` | **方向 A**(与 MLX 同,源码 line 46-52 印证) |
| fla `chunk_rwkv7`(RWKV-PEFT 训练用) | `S[i,j] += k[i]·v[j]` | `y = r^T @ S` | **方向 B**(A 的转置) |

**关键(WRWV-Runner rwkv.py:843-857 的真实分支)**:Runner 按 model.version 分两条加载路径:

```python
if model.model.version >= 7:          # x070
    state_tuned[i*3+1] = time_state.to(float)        # 原样, 不 transpose
else:                                  # v5/v6
    state_tuned[i*3+1] = time_state.transpose(1,2)   # 转置
```

我们的模型是 x070,Runner 判 version=7,**不 transpose**。因此导出器存原样训练方向即可。

> v5/v6 的 `transpose(1,2)` 是为把 fla 方向(PEFT 训练输出)转成 CUDA 方向。x070 不需要。

### 2.2 导出格式(对照源码确认)

| 项 | 值 | 来源 |
|---|---|---|
| 容器 | `torch.save(OrderedDict, path)` | RWKV-PEFT `trainer.py` |
| 键名 | `blocks.{i}.att.time_state` | RWKV-PEFT `att.py: RWKV_Tmix_x070_State` |
| 形状 | `(H, D, D)` = `(num_heads, head_dim, head_dim)` | 同上 |
| dtype | fp32(`torch.float32`) | Runner 加载时 `.to(torch.float)` |
| 转置 | **x070: 存原样(不 swapaxes)**;v5/v6 兼容:`export_pth(x070=False)` 存 swapaxes | RWKV-Runner rwkv.py:843 |

### 2.3 验证结果

```
① round-trip 数值:  导出 pth → torch.load 读回 → allclose 原始 state (x070 原样)
                    0.4B: max diff 0.00e+00

② 键名/形状/dtype:  24 层全是 blocks.{i}.att.time_state, shape (H,64,64), fp32 ✅

③ Mac 挂载等价性:   pth 注入 MLX generate 的输出 == 训练 state 直接注入的输出
                    (逐字符一致) — 注:仅证明 Mac 内部自洽

④ Windows 真机验收: noswap 版输出英文+关键词命中(方向正确);swap 版输出碎渣(方向错)
                    官方 roleplay state 同链路正常 — Runner v7 state 路径健康
```

> **重要教训**:①②③ 都在 Mac/MLX 侧闭环,只能证明"内部自洽",**不能证明与外部消费者兼容**。
> ④ 的 Windows 真机验收才是导出格式的最终判据。初版误把 ③ 当作端到端证明,导致方向错误未被发现。

## 三、训练循环产品化(任务②)

### 3.1 落地特性

| 特性 | 实现 | P0 修正点 |
|---|---|---|
| lr 默认值 | **0.01**(可调) | P0 实测 1.0 导致 state 爆炸(std 7~13),0.01 才健康 |
| state std 监控 | 每 epoch 算均值,>1.0 发 `std_warning` 事件(不中断) | P0 报告 §三.1 建议 |
| held-out 早停 | 每 epoch 在 held-out 算 loss,连续 patience 次不改善则停 | P0 报告 §七.1 建议(废除固定 epoch) |
| checkpoint/恢复 | state + Adam(m/v) + 进度持久化,`--resume` 断点续训 | 新增 |
| 结构化事件 | JSON lines(start/step/epoch_end/std_warning/checkpoint/early_stop/final) | 为 Phase 3 sidecar IPC 铺路 |

### 3.2 端到端训练验证(0.4B,2 epoch)

CLI 端到端跑通,事件流结构正确:

```json
{"type": "start", "config": {"lr": 0.01, "epochs": 2, "early_stop": true, ...}}
{"type": "epoch_end", "epoch": 0, "loss": 5.247, "state_std": 0.1055, "held_out_loss": 5.442, "patience_left": 3}
{"type": "epoch_end", "epoch": 1, "loss": 3.316, "state_std": 0.1073, "held_out_loss": 5.436, "patience_left": 3}
{"type": "checkpoint", "epoch": 1, "path": ".../epoch001.npz"}
{"type": "final", "best": 5.436, "elapsed": 37.7}
```

中断恢复验证:`--resume epoch001.npz` 正确从 epoch 2 继续(只跑了 1 个 epoch)。

### 3.3 早停验证(1.5B,充分训练)

1.5B 完整训练(20 epoch 上限,patience=3)的早停行为:

```
ep0: train_loss=2.716 held_out=3.776 std=0.1267 patience=3 ★ BEST
ep1: train_loss=1.459 held_out=2.740 std=0.1436 patience=3 ★ BEST
ep2: train_loss=0.714 held_out=2.984 std=0.1589 patience=2
ep3: train_loss=0.382 held_out=3.233 std=0.1737 patience=1
ep4: train_loss=0.153 held_out=3.423 std=0.1828 patience=0
早停 @ ep4, best_held_out=2.740 (ep1)
```

**观察**:train_loss 持续下降(2.7→0.15)但 held-out 从 ep1 后持续上升——典型过拟合。
早停正确地在 ep1 的 state 上停住(best_states 保存的是 ep1)。state std 0.14~0.18,
全程在健康区间(< 1.0),无爆炸。这正是 P0 报告 §七.1 建议的"废除固定 epoch + 加早停"
的实际效果。

## 六、0.4B vs 1.5B 翻译效果对比(重点验证)

### 6.1 实验设置

| 模型 | state | 训练程度 | state std |
|---|---|---|---|
| 0.4B | ep04(P0 验收通过) | 充分训练 | 0.0886 |
| 1.5B | 早停 best(ep1) | 充分训练(held-out 2.74) | 0.1255 |

两个 state 都训练到收敛(非欠拟合),state std 都在健康区间。用相同的句子集和配置对比。

### 6.2 训练集内(T1-T5,模型见过)— 1.5B 明显更好

| # | 中文核心 | 0.4B 输出 | 1.5B 输出 | 胜出 |
|---|---|---|---|---|
| T1 | 天气好/散步 | The weather is really nice today, perfect for going out for a walk.(逐字背诵) | **The weather is really nice today, perfect for going out for a walk.**(逐字,同) | 平 |
| T3 | 拿铁/香浓 | This coffee is very rich, and it's very **acidic**; I recommend you try it(错词) | **This latte at the café is very rich and aromatic**; I recommend you try it(aromatic 正确) | **1.5B** |
| T5 | 预算/原材料涨价 | The budget...is very **expensive**; the price...should be **increased**(走样) | **The budget for this project needs to be adjusted because the price of raw materials has increased**(准确完整) | **1.5B** |

**典型对比 — T3「拿铁香浓」:**
- 0.4B 把 aromatic(香浓)说成 **acidic**(酸的)—— 语义错误
- 1.5B 准确输出 **"very rich and aromatic"** —— 接近参考译文,且 "latte at the café" 比 0.4B 的 "This coffee" 更精确

**典型对比 — T5「原材料涨价」:**
- 0.4B 走样成 "very expensive"(很贵),丢失了"调整/原材料"语义
- 1.5B **"needs to be adjusted because the price of raw materials has increased"** —— 语义准确、语法完整,接近参考译文

**推论**:1.5B 的更大模型容量让 state 能固化更精确的词级映射和语法结构。训练集内
(记忆/复现场景)质量提升是实打实的收益。

### 6.3 测试集(S1-S5,模型未见过)— 1.5B 更不稳定

| # | 中文核心 | 0.4B 输出 | 1.5B 输出 | 胜出 |
|---|---|---|---|---|
| S1 | 开会/带文档 | Please remember to **open the meeting**...**bring**...reading(走样但抓核心) | He **forgot to work on the project file** yesterday afternoon(抓"文档/忘",语法流畅) | 各有千秋 |
| S2 | 降雨/内涝 | **Due to the rains**, the rains are light...(抓住"下雨") | Due to the **raindrop**...raindrop...raindrop...(重复退化) | **0.4B** |
| S4 | 合同/审阅 | Please **confirm this contract**...(抓住"合同") | Please confirm this **meeting** to the first three minutest...(重复退化) | **0.4B** |

**典型对比 — S2「降雨内涝」:**
- 0.4B:**"Due to the rains"** —— 抓住了"因为下雨"的核心语义(虽然后半句走样)
- 1.5B:**"raindrop...raindrop...raindrop..."** —— 陷入重复退化,语义丢失

**典型对比 — S4「合同审阅」:**
- 0.4B:**"Please confirm this contract"** —— 抓住了"合同/确认"
- 1.5B:**"Please confirm this meeting to the first three minutest"** —— 把 contract 说成 meeting,且 "minutest" 重复

**推论**:1.5B 在测试集(未见过)上泛化时**更不稳定**,容易陷入重复退化。
这**不是 P1 的 bug**,而是 state tuning 在不同模型规模上的特性差异:
- 0.4B 学得"温和但稳" —— state std 0.09,泛化时行为保守
- 1.5B 学得"精但脆" —— state std 0.13(更大),泛化时容易在概率接近处震荡/退化

可能的解释:1.5B 容量大,同样的 lr 下 state 长得更大(std 0.13 vs 0.09),更强的初始
偏置在训练分布内表现好,但偏离训练分布时更容易把模型"推入"退化的重复模式。

### 6.4 一个对照:1.5B 欠训练(2 epoch)的效果

为对比,先用只训 2 epoch 的 1.5B state(state std 0.13,held-out 3.53,**未收敛**)跑
同一批句子,看欠训练的影响:

| # | 1.5B 充分训练(ep1) | 1.5B 欠训练(2ep) |
|---|---|---|
| T3 | This latte...very rich and **aromatic** ✓ | This coffee is very rich and **aromatic** ✓ |
| T5 | ...price of raw materials has **increased** ✓ | This project needs to be **reorganized**... ✗(走样) |
| S2 | raindrop...raindrop(退化) | Please connect the parts of this **computer network**(完全跑偏) |
| S5 | (重复退化) | This product is very popular among the **male population**(胡说) |

**推论**:欠训练的 1.5B 在训练集内已有一定效果(T3 对),但测试集语义漂移严重
(S2 说成 computer network、S5 说成 male population)—— state 还没固化稳定的
"中→英翻译"模式。这印证了早停的价值:在 ep1(held-out 最佳)停住,比训到 ep4
(train_loss 更低但已过拟合)泛化更好。

## 七、1.5B 内存实测(任务⑤)

### 7.1 实测数据(M5 16GB,ctx 512,bsz 1,bf16)

| 模型 | 权重 | 训练峰值 | 每步耗时 | state 大小 | 16GB |
|---|---|---|---|---|---|
| 0.4B | ~0.9GB | **1.39GB** | 0.18s | 6.0MB(24层×16头) | ✅ 舒适 |
| 1.5B | ~3.0GB | **4.00GB** | 0.29s | 12.6MB(24层×32头) | ✅ 舒适 |

**关键发现**:1.5B 峰值仅 4.00GB,**远低于 P0 预估的 8~11GB**(预估偏保守)。
16GB M5 上完全舒适运行,**无需 gradient checkpointing**。

### 7.2 v1 支持档位结论

- 0.4B + 1.5B:**官方支持**(16GB 舒适)
- 2.9B:建议标注"24GB+",16GB 边缘(未实测,预估 >10GB)

## 十、给 Phase 2/3 的建议

1. **1.5B 泛化不稳定**:测试集重复退化(§六.3)可能是 lr 或 state std 偏大所致。
   Phase 2 可实验:1.5B 用更小 lr(0.005)或更严的 max_state_std(0.5),看泛化是否更稳。
2. **早停指标**:当前用 held-out loss 早停,但翻译任务的真正指标是"语义相关性"。
   Phase 2 可考虑加一个轻量行为指标(如英文占比)作为辅助早停信号。
3. **预览优先**:Phase 2 的内置预览,A/B 对比(有/无 state)已在 CLI 的 `preview --ab`
   实现,直接复用即可。1.5B 训练集内的质量提升(T3/T5)是预览的卖点。
4. **torch 切割**:当前 torch 是必选依赖(~2GB),Phase 3 打包进 .app 时体积偏大。
   可用手搓零依赖 pickle 导出器(legacy tar 格式)切割,作为独立优化项。
5. **sidecar IPC**:事件流(events.py)已是 JSON lines,Phase 3 的 sidecar 协议
   直接消费此事件流驱动 SwiftUI 进度面板/loss 曲线,无需重新设计。
