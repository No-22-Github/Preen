# P0-04：会话重置与跨面板安全确认

> 优先级：P0  
> 状态：已评审，待实施  
> 所属版本：v1.1.0

## 一、问题

对话 toolbar 中加载 State、卸下 State、断开连接已经会在存在聊天记录时确认。但从训练完成页或训练记录点击“去对话”会直接切换 State 并清空已有会话，绕过了相同的安全规则。

同一后果在不同入口表现不同，会让用户无法建立稳定预期。

## 二、目标

- 所有会清空或替换会话的入口使用同一决策逻辑。
- 在真正发生清空前确认，而不是提前打扰。
- 空会话保持一键直达。
- 用户取消后，模型、State、页面和会话均保持原样。

## 三、需要统一拦截的动作

- 加载或替换 State。
- 卸下 State。
- 改变 template / reasoning / think。
- 切换模型。
- 断开推理后端。
- 从训练完成页进入比较或对话。
- 从训练记录进入比较或对话。
- 打开另一个训练记录对应的 State。

## 四、方案

将确认状态从 `ContentView` 的个别 toolbar 动作提升为 App 级会话替换请求：

```text
requestSessionReplacement(intent)
  ├─ 无消息：立即执行
  └─ 有消息：展示确认 → 执行或取消
```

`intent` 至少包含目标模型、目标 State、目标会话配置和完成后的页面。弹窗前先做不会改变会话的路径、格式与 State 结构预检；用户确认后再统一停止生成、结束旧会话并启动新会话。

单模型常驻架构下不承诺跨模型切换的原子回滚：用户确认后，旧会话已结束；若新模型或新会话启动失败，App 进入断开态、保留可诊断错误并允许重试，不伪装已恢复旧 cache。

## 五、文案

标题必须描述具体动作，例如：

- “比较这个 State 会结束当前会话？”
- “切换模型会结束当前会话？”
- “更改模板会清空当前会话？”

说明文字包含：

- 哪些内容会丢失。
- 新会话将使用什么模型 / State。
- 此操作不可撤销。

破坏性按钮使用具体动词，不统一写“继续”。

## 六、验收标准

- [ ] 复现“已有一轮聊天 → 历史记录 → 比较效果”，必须出现确认。
- [ ] 点击取消后原消息、模型、State、selection 不变化。
- [ ] 点击确认后只清空一次，不出现中间空白 session 或重复连接。
- [ ] 空会话从完成页进入比较不弹确认。
- [ ] toolbar 与跨面板入口使用相同确认组件和后果判断。
- [ ] 正在生成时请求替换会话，先展示包含“将停止当前生成”后果的确认；只有确认后才 abort 并替换，取消后当前生成继续。
- [ ] 确认后新会话启动失败时，页面明确进入断开/失败态，不展示虚假的旧会话恢复。
- [ ] VoiceOver 能读出动作后果，默认焦点落在取消或非破坏性动作。

## 七、不做

- 不在本需求实现完整会话持久化或恢复。
- 不增加“以后不再提醒”。
- 不用可撤销假象掩盖后端 cache 无法回滚的事实。
