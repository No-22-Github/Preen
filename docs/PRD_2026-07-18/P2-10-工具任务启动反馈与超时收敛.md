# P2-10：工具任务启动反馈与超时收敛

> 优先级：P2  
> 状态：已评审，待实施  
> 建议版本：v1.1.x

## 一、问题

数据预览等一次性工具首次运行时，需要启动内嵌 Python、导入依赖并加载 tokenizer。界面可能较长时间停在“正在启动检查进程…”，用户无法判断是在正常冷启动、等待资源，还是已经卡死。

当前任务最终能够完成，但启动阶段信息不足。

## 二、目标

- 区分启动、加载 tokenizer、检查、渲染、收尾阶段。
- 在进程异常退出或长期无事件时自动收敛到可重试状态。
- 取消动作始终有终态，不永久停留在“正在取消”。

## 三、阶段反馈

统一离线工具状态：

```text
preparing → starting_process → loading → processing → writing → completed
                                               └→ cancelling → cancelled
                                               └→ failed
```

界面文案示例：

- 正在启动内嵌 Python…
- 正在加载 tokenizer，首次运行可能较慢…
- 正在检查第 120 / 300 条…
- 正在生成预览…

只有有真实 total 时使用确定进度条；启动和加载阶段使用不定进度。

## 四、超时与进程收敛

- 进程启动后在合理时间内没有 `started` 事件，转入“启动时间较长”提示，但不立即判失败。
- 超时只针对“该阶段应有事件却长时间无任何心跳/进度/终端事件”的不活跃时长，不对模型转换、量化或大数据检查设全任务总时长上限。
- 达到当前阶段锁定的无心跳硬超时时，终止子进程并由 Swift 侧补发结构化 failed 终态；不要求已被终止的子进程自己回事件。
- 启动、加载与 processing 可有不同的不活跃阈值；具体秒数在实施前通过 macOS 14 / 26 冷启动及长任务实测锁定，不能看完数据后临时修改判据。
- cancel 后等待 SIGINT 收尾；超过宽限期再终止，并始终生成 cancelled 终态。

## 五、诊断

失败状态提供：

- 重试。
- 查看诊断日志。
- 复制错误摘要。

stderr 不直接铺满主页面，保留在日志入口。

## 六、验收标准

- [ ] 冷启动时用户能看见当前阶段和“首次可能较慢”的提示。
- [ ] 300 条示例数据检查能展示实际处理进度。
- [ ] 模拟进程无法启动时，界面在有限时间内进入 failed。
- [ ] 模拟无心跳/无 stdout 事件时，达到当前阶段预设的不活跃硬超时后可重试。
- [ ] 模拟持续产生合法进度的长时间转换任务，不因总时长较长被误杀。
- [ ] cancel 最终进入 cancelled，不永久停留在 cancelling。
- [ ] 失败与取消都不会留下孤儿 Python 进程。

## 七、不做

- 不伪造百分比。
- 不用固定动画假装任务有进展。
- 不在本需求优化 tokenizer 或模型加载性能。
