# P0 理论指南:在 MLX 上做 RWKV-7 State Tuning

本文只讲原理,不讲操作。目的是让你在读 Agent 写出的代码时,能判断每一处实现是对是错、为什么。

---

## 一、State Tuning 在数学上到底在训什么

RWKV-7 的每一层 time mixing 维护一个矩阵值状态。对每个注意力头,state 是一个 64×64 的矩阵 S,随序列逐 token 演化。RWKV-7 的更新规则(delta rule 变体)可以写成:

S_t = S_{t-1} · diag(w_t) + S_{t-1} · a_t · b_t^T + v_t · k_t^T

其中 w 是逐通道衰减,a、b 来自 in-context learning rate 的低秩投影,v·k^T 是本 token 写入的新信息。输出 y_t = S_t · r_t。直观理解:S 是一块会衰减、会被选择性擦写的"快速权重"内存,序列里发生过的一切都被压缩在这个矩阵里。

标准推理时 S_0 = 0,即模型从"一无所知"的空白状态启动。State tuning 做的事情是:**把 S_0 从零向量变成可训练参数,用梯度下降找一个初始状态,使模型从这个状态启动时的行为符合训练数据的分布**。模型的所有权重(投影矩阵、LoRA、LayerNorm、embedding)全部冻结,唯一被优化的就是每层每头的这个 64×64 矩阵。

这解释了 state tuning 的能力边界。一段 prompt 经过前向传播后留下的 state,和一个训练出来的 state,在数学上是同一种对象——所以 state tuning 本质上等价于"寻找一个最优的虚拟前缀",它能固化的东西和 prompt 能表达的东西属于同一量级:说话风格、任务模式(比如"看到中文就翻译成英文")、角色设定。它固化不了新知识,因为知识存在冻结的权重里。你的验收目标"不给 prompt 也自主翻译"恰好落在它能力圈的正中心,这也是为什么这个验收标准是合理的。

参数量核算:0.4B 模型 24 层、hidden 1024、head_dim 64,即每层 16 个头。可训练参数 = 24 × 16 × 64 × 64 ≈ 157 万个数,fp32 下约 6MB。相对 4 亿冻结参数,你在优化一个极小的自由度空间——这既是它训得快、不易灾难性遗忘的原因,也是它表达力有限的原因。

## 二、为什么梯度只能走 ops 路径

mlx-lm 的 rwkv7 实现里,wkv 递归有两个等价的前向:一个手写 Metal kernel,一个纯 MLX ops 的逐 token 循环。数值上它们算的是同一个东西,但对自动微分而言完全不同。

MLX 的自动微分靠对计算图里每个原语算子应用已知的 VJP(向量-雅可比积)规则。`mx.fast.metal_kernel` 创建的是一个黑盒:MLX 只知道输入输出形状,不知道内部数学,除非作者手动注册 custom VJP——这份实现没有注册。所以反向传播到 kernel 这里,梯度链条直接断裂。而纯 ops 路径由矩阵乘、逐元素乘加、stack 这些原语组成,每一步都有 VJP,梯度可以一路穿透 512 个时间步回到 S_0。

这带来一个必须理解的验证要点:**梯度冒烟测试查的不是"有没有报错",而是"梯度是否非零"**。断裂的梯度链在 MLX 里往往不报错,只是返回零梯度——训练会"正常"运行,loss 纹丝不动。所以冒烟测试的正确姿势是逐层检查 state 梯度的范数,任何一层是零就说明 patch 没生效或 state 没进计算图。

关于 `mx.compile` 和 grad 的交互:MLX 的 compile 是可微分变换,理论上 compile 过的函数能正常求导。但这是 P0 里唯一存在"理论上没问题、实践上可能有边角 bug"的地方,所以 roadmap 里的兜底方案是去掉装饰器裸跑——慢三五倍,但排除一个变量。

反向传播的内存代价也源于这个循环结构:逐 token 前向意味着 autodiff 需要保留全部 512 步的中间 state(每步 B×16×64×64)和各投影的中间量,用于反向时的链式法则。这就是训练峰值内存远超权重体积的原因,也是 gradient checkpointing(丢弃中间量、反向时重算)能大幅省内存的原理——拿计算换存储,而不是拿硬盘换存储。

## 三、训练超参为什么长成那个反直觉的样子

**学习率 1.0**。常规微调 lr 在 1e-5 到 1e-4 量级,state tuning 的 lr 从 1.0 起步,大了四五个数量级,这不是笔误。原因有三层。第一,被优化的 state 不是权重,它的数值分布和权重完全不同——推理时 state 元素的典型幅度可以到 O(1) 甚至更大,而参数初始为零,要从 0 走到工作幅度,小学习率意味着永远走不到。第二,只有几 MB 参数、几十条数据,不存在常规微调里"大 lr 破坏预训练知识"的风险,权重全冻着。第三,loss 对 S_0 的曲面相对平缓(它经过 512 步衰减和擦写才影响输出,梯度被稀释),需要大步长补偿。配套的 cosine 衰减到 0.01 和短 warmup 是为了后期收敛稳定。

**ctx 512 而不是越长越好**。看更新公式:S 每一步都乘 diag(w),w < 1,这意味着 S_0 的影响随序列长度指数衰减。序列越长,末端 token 的 loss 对 S_0 的梯度越微弱,训练信号越稀。同时你的目标行为是"冷启动即翻译"——用户真实使用时输入就是一两句话,训练分布和使用分布应当一致。短 ctx 既是优化上的正确,也是任务上的正确。

**每条样本必须从可训练 state 独立启动**。这是 P0 最容易被 Agent 写错的地方。如果把多条样本拼接成长序列,只有第一条真正从 S_0 出发,后面的样本都从"前一条样本留下的 state"出发——你训出来的 S_0 学到的是"对话进行中的续写模式",而非"空白启动即翻译"。正确做法是 batch 内每条序列都以 S_0(广播)为初始,padding 对齐,pad 位置 loss 置零。

**loss mask 只算 Assistant 段**。翻译对里 User 段(中文原文)是条件,不是学习目标。如果对 User 段也算 loss,你在训练 state "生成中文句子",这会稀释甚至污染翻译信号。mask 的原理就是把交叉熵按位置加权:条件部分权重 0,目标部分权重 1。

**epoch 数与停止**。90 条高一致性语料、157 万参数,10 到 20 个 epoch 内 loss 会进入平台期。State tuning 的过拟合表现不是背答案,而是行为僵化(输出模板化、无视输入内容),所以停止依据是 held-out 行为而不是 train loss 压到多低。

## 四、验证的三级理论:为什么是这个顺序

三级验证各自排除一类假设,顺序不可跳。

**第一级,梯度冒烟**:回答"计算图是否连通"。一个 batch、一步反向,逐层查 grad 范数非零、无 NaN/Inf。它排除的是结构性错误:patch 未生效、state 未注入、dtype 造成的静默截断。

**第二级,微型过拟合**:回答"优化管线是否正确"。10 条固定样本训到 loss < 0.5。原理是:一个正确的训练管线必然能记住 10 条数据——157 万参数记 10 个短序列的容量绰绰有余。如果连过拟合都做不到,问题必在管线(mask 错位、label 偏移、lr 调度错),而不在数据或模型。这一步的价值在于它是**充分性测试**:通过了才有资格谈泛化。

**第三级,held-out 行为验收**:回答"学到的是行为还是背诵"。用未训练的 10 条 + 若干分布外中文(口语、数字、专名)测试,标准是"是否表现出翻译行为",不是"翻译质量"。0.4B 的翻译上限本来就低,你在验证 state 固化了任务模式,BLEU 分不是 P0 的问题。定量线:10 条里 ≥8 条输出为英文翻译尝试。

## 五、跨实现对照:llama.cpp 作为参考真值

你担心 Agent 写的 Python 推理有"暗病",这个担心的正确解法不是比速度,而是**比输出**。原理:同一份权重、同一个输入、贪心解码(温度 0),两个独立实现应当给出高度一致的输出。llama.cpp 的 RWKV-7 实现经过社区大量使用,可以当作参考真值(oracle),用来交叉验证 MLX 路径的数值正确性。

做这件事有三个前提,缺一个对照就失去意义。第一,**权重精度必须对齐**:MLX 侧 bf16/fp16,llama.cpp 侧必须用 fp16 的 GGUF,不能拿 q4 量化版——量化误差远大于实现 bug 的信号,会把你要找的暗病淹没。第二,**采样必须关闭**:贪心解码,任何随机性都让输出不可比。第三,**tokenizer 一致性先单独验证**:两边对同一句中文的 token 序列应当完全一致,若 token 都不同,后面的比较是鸡同鸭讲。

比较的粒度分两档。粗档:贪心生成 50 token,前 10~20 个 token 应当完全一致或极早分叉(fp16 累加顺序不同导致的合法分叉存在,但通常出现在概率接近的分叉点,且分叉后语义仍应相近)。细档:取若干位置的 logits 做数值 diff,同一 token 的 logit 差应在 1e-2 量级内(fp16 精度下的合理浮动)。粗档不过直接说明有 bug;细档能定位 bug 大概在哪一侧。

顺带说,ops 路径与 Metal kernel 路径之间也应做同样的自我对照:同输入下两条路径的输出应在 fp16 容差内一致。这个内部对照零成本,且能单独验证"训练用的前向"和"推理用的前向"算的是同一个函数——这对你的工具至关重要,因为你训练时走 ops、预览时走 kernel,两者不等价的话训练结果就不可信。

至于速度对比,它回答的是另一个问题("裸推理谁快"),与正确性无关,与 Phase 2 的架构选择也无关(llama.cpp 挂不了外部 state,已被功能性一票否决)。有兴致可以顺手记录,没兴致不做,你的判断"不做也行"是对的——但输出对照建议保留,它才是这套动作里真正值钱的部分。

## 六、导出与挂载的格式理论

训练产物要被 RWKV Runner / Ai00 消费,而它们期待的是 RWKV-PEFT 生态的 `.pth` 格式:一个 PyTorch pickle,键为逐层的 state 名称,值为该层的 state 张量。这里有两个必须对齐的维度。

第一,**键名与形状约定**。RWKV 原生权重命名体系(blocks.N.att.*)和 HF/fla 命名体系(model.layers.N.attn.*)是两套平行世界,state 文件属于前者。张量形状上要注意:训练时你的参数带 batch 维(1, H, D, D)且可能是 bf16,导出时通常要去掉 batch 维并转 fp32/fp16——具体以 RWKV-PEFT 导出代码为准,这是 P0 之后唯一需要"抄作业对答案"的地方。

第二,**数学语义对齐**,这是比键名更隐蔽的坑:不同实现对 state 矩阵的行列约定可能互为转置(S·r 还是 r·S,k·v^T 还是 v·k^T)。如果 MLX 实现和 RWKV 原生实现的约定相反,导出时不转置,挂载后模型行为会彻底错乱而不报任何错。验证方法是行为级的:训一个特征极明显的 state(比如满嘴 emoji),挂到 Runner 里,行为出现即约定正确;行为怪异则先试转置再排查其他。

## 七、P0 的完成定义

理论上,P0 结束时你应当能对以下每个命题给出"是"的回答,并且知道依据是什么:梯度可以从 loss 穿透 512 步递归抵达每一层的 S_0;优化器能驱动这组参数把 10 条样本的 loss 压到接近零;90 条训练后,模型对未见中文表现出无提示翻译行为;MLX 的两条前向路径在容差内等价;MLX 与 llama.cpp 对同一权重给出一致输出。五个命题全真,后端成熟度即告确认,剩下的全部是产品工程。
