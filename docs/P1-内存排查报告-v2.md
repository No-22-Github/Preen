# P1 训练内存排查报告 v2

> **生成时间**:2026-07-10
> **排查对象**:RWKV-7 state tuning 训练内存(cache 膨胀)异常
> **状态**:排查完成。本轮最大发现:**所有"无限档"cache 稳态值都被 GPU working_set 软上限削顶**,此前所有按"真稳态"做的结论需以此修正。

## 0. 结论先行

1. **没有代码回归**。EOS 修复(`44cca39`)只动 `generate()` 推理路径,训练循环未触及。
2. **cache 稳态被 GPU 硬上限削顶**——`active + cache` 精确顶在 `max_recommended_working_set_size`(12.71G)的 95%。0.4B 和 1.5B 无一例外。**此前所有"无限档 cache 稳态值"(0.4B=11.15G,1.5B=8.97G)都是削顶值,真实需求未知。**
3. **cache 增长的驱动量是数据中最长样本的 token 数**——逐样本 trace 铁证:cache 只在长样本出现时跳变,短样本 Δcache≈0,池子只增不缩。
4. **可用的缓解手段**:`mx.set_cache_limit` 立即生效且零代码代价(§6 选项 B)。

## 1. 自变量判决(任务1)

### 受控实验(0.4B,100 步,无限 cache)

| 组 | 构造方法 | mean | max | p95 | 稳态 cache | 可信 |
|---|---|---|---|---|---|---|
| L64 | 筛 length∈[54,74] | 67.6 | 69 | 69 | 1.04G | ✓真实 |
| L128 | 筛 [109,147] | 117 | 133 | 133 | 4.38G | ✓真实 |
| L256 | 筛 [218,294] | 253 | 261 | 261 | 8.51G | ✓真实 |
| mixed | 95%短(≤50)+5%长(≥250) | 64.5 | 432 | 279 | 8.63G | ✓真实 |
| smoke200 | NekoQA 前 200 条 | 113 | 273 | 193 | 11.15G | ⚠️**削顶** |

### 逐样本 trace 铁证(mixed 集)

```
step 0-8   (全 50tok): cache 0.48→0.51G, Δ≈0      短样本不撑池子
step 9     (331tok):   cache 0.51→3.49G, Δ+2.98   长样本跳变
step 10-25 (全 50tok): cache 3.49G 纹丝不动        池子只增不缩
step 71    (402tok):   cache 6.27→8.63G            锁定,之后28短样本无法降低
```

**cache 单调递增(容差 0.01G 内绝不下降),跳变只发生在长样本出现时。** 这是 MLX Metal allocator 的设计——释放的 buffer 留池复用,不还给系统。

### 拟合(等长集 3 点,注明适用边界)

```
cache ≈ 0.038 × max_token − 1.2    [G]   RMS = 0.39G
适用:0.4B,等长或近等长数据,max∈[70,260]
禁止外推:max>260 无实测支撑
```

**公式的局限(诚实交代):**

| 数据点 | max | 实测 cache | 公式预测 | 说明 |
|---|---|---|---|---|
| L64/L128/L256 | 69/133/261 | 拟合点 | — | 等长,公式吻合 |
| mixed | 432 | 8.63G | 15.22G | **偏离 -6.6G** |
| smoke200 | 273 | 11.15G | 9.17G | **削顶值**,偏离无意义 |

mixed 的偏离揭示:**cache 不只取决于数据集的 max,还取决于长样本是否真正在训练中被"见过"**。mixed 只有 5% 长样本,100 步里最长样本(402tok)只在 step71 出现一次,撑到 8.63G 后没再遇到更长的去继续撑。因此公式不能外推——混合密度数据的真实需求取决于"实际见过的最长样本"和曝光次数,无法用单一 max 刻画。

### 判决(一句话)

> **P4 预检应读数据的 max token 长度。** 逐样本 trace 证明 cache 只被长样本撑大。但注意:在 16GB 机器上,无论数据多长,cache 都会被 working_set 上限削顶在 ~11G(0.4B)/~9G(1.5B),不会无限增长——这是 OOM 的保护机制,但也是"真实需求不可测"的原因。

## 2. 削顶假设验证(决定性发现)

### GPU 硬限制(`mx.metal.device_info()` 真理值)

```
max_recommended_working_set_size = 12.71 G   (总 working set 软上限)
max_buffer_length                =  9.54 G   (单个 Metal buffer 上限)
memory_size                      = 16.00 G
```

### active + cache = 95% working_set(两个模型都精确顶住)

实测 80-100 步稳态(三口径互斥:active = 当前活跃计算,cache = 回收池,两者不重叠):

| 模型 | active | cache | **active+cache** | 占 working_set |
|---|---|---|---|---|
| 0.4B | 0.93G | 11.15G | **12.08G** | **95%** |
| 1.5B | 3.11G | 8.96G | **12.07G** | **95%** |

**两个模型的 active+cache 都精确顶在 12.07-12.08G。** MLX 在接近 working_set 软上限(12.71G)的 95% 时停止扩大 cache 池。

### 这解释了所有此前的"反常"

| 此前观察 | 削顶解释 |
|---|---|
| 1.5B cache(8.97G)比 0.4B(11.15G)"低" | 不是真低。1.5B 的 active(3.11G)>0.4B(0.93G),挤占了 cache 空间。差值 2.18G ≈ active 差 2.18G,完全对账 |
| smoke200(max273)→11.15G vs L256(max261)→8.51G 差 2.6G | smoke200 被削顶在 11.15,真实需求未知;L256(8.51)未削顶。两者不可直接比 |
| "步数是隐藏变量" | ❌ 错误假设(已撤回)。真正变量是削顶线 |

### 1.5B 数据可信度

| 实验 | cache | 判定 |
|---|---|---|
| L40_inf | 0.86G | ✓真实(远低于削顶线 8.9G) |
| nekoqa200_inf | 8.97G | ⚠️**削顶值**(真实需求 ≥8.97G) |
| L256_inf | 8.97G | ⚠️**削顶值**(真实需求 ≥8.97G) |

## 4. cache_limit 扫描(任务3,1.5B nekoqa200,100 步同口径)

**基准改为无限档**(修正上轮错误):

| cache_limit | 稳态 cache | compressor | ms/step(末10) | 内存压力 |
|---|---|---|---|---|
| 4G | 4.00G | **1.45G** | 1242 | 🟢绿 |
| 6G | 6.00G | 4.64G | 1117 | 🟡黄 |
| 8G | 8.00G | 4.29G | 1775 | 🟡黄 |
| 10G | ~9.0G(中止) | 5.28G | 6-12s | 🔴换页 |
| 无限(基准) | ≥8.97G(削顶) | 4.41G | **2552** | 🟡黄 |

c10G 因系统重度换页极慢(6-12s/step),跑到 step50 中止——"跑不动"本身是有效数据。

### 速度结论(重写)

- **c4G 是"最快的绿档"**:compressor 1.45G(唯一压力绿),1242ms/step。
- 相对无限档基准(2552ms):c4G **快 51%**,c6G 快 56%,c8G 快 31%。
- **"中间值最慢"现象不成立**(0.4B 上轮的偶发观察已修正)。1.5B 上速度随 cache_limit 非单调,规律是:cache_limit 接近自然需求时最快(少重分配),过高被系统换页拖慢。
- **1.5B 在 16GB 上压力绿的最大 cache_limit = 4G**。

### 速度对比修正说明

上轮拿 80 步(c4/c6/c8)比 150 步(无限)的 ms/step,口径不一致,已作废。本轮全部 100 步同口径(无限档基准 2552ms 取自 nekoqa200_inf 80 步末10,口径略异但量级可信)。

## 5. 16GB 容量判决表

### GPU 硬上限(真理值)

```
max_recommended_working_set_size = 12.71G  →  削顶线 = 12.71×0.95 = 12.07G
  cache 自然上限 = 12.07 − active
  0.4B(active~0.9G): cache 削顶在 ~11.1G
  1.5B(active~3.1G): cache 削顶在 ~8.9G
```

压力颜色:compressor <2G=🟢,2-4G=🟡,>4G=🔴

### 0.4B(H=16)

| 数据 max token | 无限 cache | 可信 | 压力 | cache_limit=4G | 可行 |
|---|---|---|---|---|---|
| ≤64 | ~1G | ✓ | 🟢 | 不需要 | ✅轻松 |
| ~130 | ~4.4G | ✓ | 🟡 | 🟢 | ✅ |
| ~260 | ~8.5G | ✓ | 🟡 | 🟢 | ✅(代价-51%耗时) |
| ~273+ | **≥11.1G(削顶)** | 削顶 | 🔴 | 🟢 | ⚠️ 设4G强制 |

### 1.5B(H=32)

| 数据 max token | 无限 cache | 可信 | 压力 | cache_limit=4G | 可行 |
|---|---|---|---|---|---|
| ≤45 | ~0.9G | ✓ | 🟢 | 不需要 | ✅轻松 |
| ~130 | 无实测 | — | — | — | 删(无数据) |
| ~260 | **≥8.9G(削顶)** | 削顶 | 🔴 | 🟢 | ⚠️设4G(代价-51%耗时,长样本慢) |

**1.5B 的 ~130 行已删除**——无法补测(任何 max>100 的 1.5B 配置都会被削顶,测不到真实值)。若需此档数据,必须在 ≥32GB 机器上测。

## 6. 缓解手段:四选项(不实现,供裁决)

### 选项 A:gradient checkpointing

| 维度 | 评估 |
|---|---|
| 理论节省 | checkpoint `_wkv7_step_ops` 循环可丢弃中间 state,机制覆盖部分(~60%)可省 |
| 预估效果 | 0.4B L256:8.5G→~3.4G;但因削顶,真实需求未知,节省量是相对"未削顶需求"而言 |
| 重算开销 | wkv7 循环完整重跑前向,单步 ~1.5-2x(1.5B nekoqa:2552ms→~4500ms) |
| mx.checkpoint 可行性 | ⚠️ `_wkv7_step_ops` 被 `@mx.compile` 装饰,checkpoint 与 compile 交互未知;Python for 循环(L 步展开)的 checkpoint 支持有限 |
| 工作量 | 中-高。改 `patch_rwkv7_for_train`,需实验 compile+checkpoint 兼容性 |
| 适用场景 | 用户必须跑长样本(max>300)且接受 2x 慢 |

### 选项 B:cache_limit 默认值

| 维度 | 评估 |
|---|---|
| 收益 | c4G 把 compressor 降到 1.45G(🟢唯一绿档),零代码改动 |
| 速度 | 相对无限档快 51%(1242 vs 2552ms) |
| 风险 | 4G 可能不够某些样本(被反复重分配),但实测 c4G 仍能跑通 |
| 工作量 | 零。一行 `mx.set_cache_limit(int(4e9))` |
| 适用场景 | **16GB 机器默认保护**,所有场景 |

### 选项 C:数据长度上限提示(P4 预检)

| 维度 | 评估 |
|---|---|
| 收益 | 零运行时成本。训练前读数据 max token,超阈值警告 |
| 判据 | 16GB:数据 max 导致预期 cache 接近削顶线(0.4B~11G/1.5B~9G)→ ⚠️警告 |
| 工作量 | 低。P4 预检加一段 token 长度统计 |
| 适用场景 | **所有场景的默认保护层**,与 A/B 不冲突 |

### 选项 D:S₀ bf16 混合精度(本轮新增)

| 维度 | 评估 |
|---|---|
| 动机 | `_wkv7_step_ops` 中间 state 全是 fp32,**根因是可训练 S₀ 是 fp32**(`make_state_params(dtype=mx.float32)`)。实测:state fp32 输入 → 循环内 y/state_out 全 fp32;MLX 提升 `bf16+fp32=fp32`。整个 wkv7 反向图的中间量因此翻倍 |
| 理论节省 | S₀ 改 bf16 → 循环中间 state 回到 bf16 → 中间量减半。机制覆盖约 60% 的部分(均为 fp32 中间 state)直接减半,**粗估可省约 30% 总 cache,但这是推算值未经实测验证**(实际节省取决于 bf16 化后其余 40% 非state中间量占比是否变化) |
| 预估效果 | 0.4B L256:8.5G→~5.9G;1.5B nekoqa:削顶线下方,可能解除削顶 |
| 数值风险 | ⚠️ state tuning 依赖 S₀ 精确累乘(wkv7 是递归 `state=state*w+v⊗k+sab`)。bf16 仅 7 位有效数字,递归 200+ 步可能累积数值误差,导致训练不稳定或 loss 不收敛。**这是主要风险,必须验证** |
| 工作量 | 低改动:`make_state_params(dtype=mx.bfloat16)` 一行。但验证成本高:需对比 fp32/bf16 的 loss 收敛曲线、final state std、导出 pth 后的推理质量 |
| 验证方案 | (1) 同数据同 seed 跑 fp32 vs bf16 各 200 步,对比 loss 曲线是否重合;(2) 检查 bf16 的 state std 是否异常漂移;(3) 导出 pth 后 Runner 推理对比风格注入效果是否退化;(4) 若 loss 差 <5% 且推理无退化 → 可选;否则否决 |
| 适用场景 | 内存压力大的长样本场景。与 checkpointing 互斥(都动 wkv7 路径),与 cache_limit 可叠加 |

**四者关系:** C 是默认保护层(必做),B 是 16GB 兜底(立即可用),A 和 D 都是"动训练路径换内存"的可选优化(D 改动小但需数值验证,A 改动大但理论上限更高)。A 与 D 互斥。

## 7. 机制验证(任务4)

### 中间 state dtype(推翻 v1 假设)

直接调 `_wkv7_step_ops(r,w,k,v,a,b,state)`:
- state fp32 输入 → **y 和 state_out 全 fp32**
- state bf16 输入 → 全 bf16
- MLX 提升:`bf16 + fp32 = fp32`

**可训练 S₀ 是 fp32,把整个 wkv7 循环提升为 fp32。v1"维持 bf16"判断错误。**

### 算术总账

```
单层单步 ≈ 3×(B,H,D,D)fp32 + 2×(B,H,D) = 776 KB
L=133 × 24 层 ≈ 2.59G  vs 实测 4.38G  → 覆盖 59%
```

| 样本 | 估算 | 实测 | 覆盖 |
|---|---|---|---|
| L69 | 1.34G | 1.04G | 129%(短样本复用) |
| L133 | 2.59G | 4.38G | 59% |
| L261 | 5.08G | 8.51G | 60% |

**机制覆盖约 60%。** 剩余 40% 是 r/w/k/v/a/b 时间混合投影、FFN、log_softmax 等反向图,未逐项标定。需求单要求"机制解释与实测差距 <30%",本轮覆盖 60%(未解释部分 40%),**未达标,诚实报告**。剩余部分的方向明确(都属训练反向图),但逐项标定需要更细的 per-op 打点,本轮未做。

### 池子行为

逐样本打点:cache 只增不缩,短样本 Δcache≈0,长样本撑大池子后永久保留。

## 8. 三口径规矩(写进项目规范)

```
GPU 硬上限(mx.metal.device_info, 真理值, 必报):
  - max_recommended_working_set_size  (总 working set 软上限)
  - max_buffer_length                 (单 buffer 上限)

每次内存汇报必给四个数:
  1. mx.get_active_memory()   当前活跃计算(与 cache 互斥)
  2. mx.get_cache_memory()    回收池(对账活动监视器的主要口径)
  3. active + cache           总占用(对比 working_set 上限,>95% 即削顶)
  4. vm_stat compressor       内存压力源(换算 ×16384/1024³)

禁止用 ps rss / ru_maxrss 报 MLX 内存(漏 GPU wired)。
测量脚本:tools/mem_probe_v2.py。测量必须跑到 ≥75 步才到稳态(或削顶)。
```
